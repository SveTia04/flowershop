## 1. Debug и Release — зачем две сборки

Пока мы (шаги `05`, `14`) собирали проект «как получится». Для сдачи нужна **релизная**
сборка. У C++ есть два основных режима:

| Режим | Что делает компилятор | Когда |
|-------|------------------------|-------|
| **Debug** | без оптимизаций, со всей отладочной информацией; легко искать ошибки | при разработке |
| **Release** | агрессивная оптимизация, без отладочного «жирка»; быстрее и меньше | для сдачи/работы |

> **Аналогия.** Debug-сборка — это **черновик с пометками на полях**: тяжёлый, но видно, где
> что. Release — **чистовик на хорошей бумаге**: компактный и быстрый, отдают заказчику.

Релизная сборка через CMake:

```bash
# Из корня проекта flowershop/. Папку build можно удалять и пересоздавать без потерь.
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release

# Готовый бинарник:
#   Linux/macOS:  ./build/flowershop
#   Windows:      build\Release\flowershop.exe
```

- `-S .` — где исходники (текущая папка), `-B build` — куда положить файлы сборки.
- `-DCMAKE_BUILD_TYPE=Release` — тот самый переключатель «чистовик». На Windows/MSVC режим
  задаётся **на этапе сборки** флагом `--config Release` (там это работает чуть иначе, чем
  на Linux, — поэтому указываем оба варианта).

## 2. Запуск сервера и конфигурация (порт, путь к БД)

Сервер не должен хранить настройки «зашитыми» в коде: на машине разработчика порт и путь к
базе одни, на сервере заказчика — другие. Поэтому настройки читаются из **конфига**
(`src/infra/config.*`, заявлен в спецификации). Учебно — простой текстовый файл
`config.ini` рядом с бинарником:

```ini
# config.ini — настройки сервера. Меняются БЕЗ перекомпиляции.
host=0.0.0.0          # 0.0.0.0 = слушать на всех сетевых интерфейсах (доступно по сети)
port=8080             # порт, на котором висит наш HTTP-сервер
db_path=flowershop.db # путь к файлу базы SQLite
log_path=flowershop.log
```

Минимальный загрузчик конфига (инфраструктура, шаг `03`):

```cpp
// src/infra/config.h
#pragma once
#include <string>

namespace fs {

struct Config {                 // структура настроек (значения по умолчанию — на случай,
    std::string host    = "0.0.0.0";   // если файла конфига нет)
    int         port    = 8080;
    std::string db_path = "flowershop.db";
    std::string log_path= "flowershop.log";

    // Прочитать настройки из ini-файла (key=value построчно).
    static Config load(const std::string& path);
};

} // namespace fs
```

В `main.cpp` конфиг применяется так:

```cpp
// src/main.cpp (фрагмент запуска)
#include "infra/config.h"
#include "infra/database.h"
#include "api/server.h"

int main(int argc, char** argv) {
    // Путь к конфигу можно передать первым аргументом: ./flowershop config.ini
    std::string cfgPath = (argc > 1) ? argv[1] : "config.ini";
    fs::Config cfg = fs::Config::load(cfgPath);

    fs::Database db(cfg.db_path);     // открыть базу по пути из конфига
    db.execScript(fs::readFile("db/schema.sql"));  // создать таблицы, если базы ещё нет
    // (на боевом сервере схему обычно заливают один раз, а не при каждом старте — см. ниже)

    fs::Server server(db, cfg);       // собрать все сервисы/контроллеры (как в шаге 09)
    std::cout << "Сервер запущен на http://" << cfg.host << ":" << cfg.port << "\n";
    server.listen(cfg.host, cfg.port);// блокирующий вызов: сервер крутится, пока не остановят
    return 0;
}
```

Запуск:

```bash
./build/flowershop config.ini      # Linux/macOS
build\Release\flowershop.exe config.ini   # Windows
# В браузере/curl: http://АДРЕС_СЕРВЕРА:8080/api/products
```

> **Тонкость про схему БД.** При самом первом запуске базы ещё нет — её надо создать из
> `db/schema.sql` + `db/seed.sql`. Но выполнять `schema.sql` при **каждом** старте плохо
> (`CREATE TABLE` упадёт, если таблица уже есть). На боевом сервере схему заливают **один
> раз** руками: `sqlite3 flowershop.db < db/schema.sql` и `sqlite3 flowershop.db < db/seed.sql`,
> а сервер потом просто открывает готовый файл. Учебно можно использовать `CREATE TABLE IF
> NOT EXISTS`, чтобы повторный запуск не ломался.

## 3. Развёртывание на Windows-сервере (ТЗ п. 3.6.3)

ТЗ требует серверное железо: **Windows 10, процессор i7, 16 ГБ ОЗУ, 500 ГБ SSD**. Хорошая
новость: наш стек **кроссплатформенный** (спецификация). Разрабатывать можно на Linux, а
итоговый бинарник собрать под Windows. Для 16 ГБ ОЗУ и SQLite-файла такой нагрузки (один
цветочный магазин) ресурсов более чем достаточно.

### 3.1. Чем собирать под Windows

Два пути, оба рабочие:

- **MSVC** (Microsoft Visual C++) — «родной» компилятор Windows. Ставится вместе с *Visual
  Studio* (можно бесплатную Community) или отдельно как *Build Tools for Visual Studio*.
  CMake его найдёт сам.
- **MinGW-w64** — порт g++ под Windows. Лёгкий, привычный тем, кто пришёл с Linux.

```bat
:: Сборка под Windows (из «x64 Native Tools Command Prompt», если MSVC):
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release
:: Результат: build\Release\flowershop.exe
```

> Все наши зависимости — header-only (`httplib.h`, `json.hpp`, `doctest.h`) плюс `sqlite3.c`,
> который компилируется **вместе с проектом** (шаг `05`). Поэтому «таскать DLL» не нужно:
> получается один самодостаточный `flowershop.exe`. Это сильно упрощает установку у заказчика.

### 3.2. Что положить на сервер

Минимальный комплект поставки (в одну папку, например `C:\flowershop\`):

```
C:\flowershop\
    flowershop.exe       (релизный бинарник)
    config.ini           (настройки: порт, путь к БД)
    db\schema.sql        (на случай первой инициализации)
    db\seed.sql
    flowershop.db         (создаётся при инициализации)
```

### 3.3. Запуск как службы Windows (кратко)

Если просто запустить `.exe` двойным кликом — он закроется вместе с окном/сессией. Чтобы
сервер работал **постоянно** и сам поднимался после перезагрузки сервера, его делают
**службой (Windows Service)**. Самый простой учебный способ — утилита **NSSM** (Non-Sucking
Service Manager), которая «оборачивает» любой `.exe` в службу:

```bat
:: Зарегистрировать наш сервер как службу с автозапуском:
nssm install Flowershop "C:\flowershop\flowershop.exe" "C:\flowershop\config.ini"
nssm set Flowershop AppDirectory "C:\flowershop"
nssm start Flowershop
:: Теперь сервер стартует автоматически при загрузке Windows и работает в фоне.
```

> Альтернатива без сторонних утилит — встроенная команда `sc create`, но NSSM проще и
> корректно перезапускает упавший процесс. Для учебного проекта достаточно знать **идею**:
> «сервер должен работать как фоновая служба, а не как окно».

### 3.4. Порт и брандмауэр (firewall)

Наш сервер слушает порт (по умолчанию `8080`). Чтобы к нему можно было подключиться **с
других компьютеров** в сети магазина, в брандмауэре Windows надо **разрешить входящие
соединения** на этот порт:

```bat
:: Открыть порт 8080 во входящих правилах брандмауэра Windows:
netsh advfirewall firewall add rule name="Flowershop 8080" ^
      dir=in action=allow protocol=TCP localport=8080
```

> **Безопасность (важно).** Сам порт `8080` отдаёт **незашифрованный HTTP**. Наружу
> (в интернет) его выставлять нельзя — только во внутренней сети. Для шифрования передачи
> (требование ТЗ п. 3.4) ставим перед сервером **обратный прокси с TLS** — раздел 6.

## 4. Резервное копирование базы (надёжность, ТЗ п. 3.5)

Вся жизнь магазина — в одном файле `flowershop.db`. Сгорел диск или файл повредился — потеряны
все заказы, остатки, история. ТЗ (п. 3.5) требует надёжность, и резервные копии (бэкапы) —
её основа.

Огромный плюс SQLite: база — это **один файл**, и бэкап = **копия этого файла**. Но копировать
файл, пока сервер в него пишет, нельзя (можно скопировать «полузапись»). Правильно — командой
`.backup` самого SQLite, которая делает **согласованную** копию даже при работающем сервере:

```bash
# Linux/macOS — согласованная резервная копия с датой в имени:
sqlite3 flowershop.db ".backup 'backup/flowershop-$(date +%F).db'"
```

```bat
:: Windows — то же самое (sqlite3.exe должен быть в PATH):
sqlite3.exe C:\flowershop\flowershop.db ".backup C:\flowershop\backup\flowershop-backup.db"
```

Это ставят на **расписание**:

- Linux — `cron` (например, каждую ночь).
- Windows — **Планировщик заданий (Task Scheduler)**: создаёте задачу, которая раз в сутки
  запускает `.bat` с командой `.backup`.

> **Правило бэкапов «3-2-1» (упомянуть заказчику).** Хранить **3** копии, на **2** разных
> носителях, **1** — вне офиса (облако/другой диск). Бэкап, лежащий на том же диске, что и
> база, не спасёт от смерти диска. Также полезно **проверять**, что копия открывается
> (`sqlite3 backup.db ".tables"`) — «непроверенный бэкап = отсутствие бэкапа».

## 5. Переход с SQLite на PostgreSQL (почему это легко)

ТЗ (п. 3.6.1) намекает на «промышленную реляционную СУБД». Мы учебно взяли SQLite, но
архитектура (шаг `03`) специально построена так, чтобы смена базы была **локальной**. Вспомним
правило: **только нижний слой (репозитории + `infra/database`) знает про конкретную СУБД**.
Сервисы, контроллеры, домен — про SQLite/PostgreSQL **ничего не знают**.

Что меняется и что нет:

| Часть проекта | Меняется при переходе на PostgreSQL? |
|---------------|--------------------------------------|
| `src/domain/*` (модели) | **Нет** |
| `src/services/*` (бизнес-логика) | **Нет** |
| `src/api/*` (контроллеры, JSON) | **Нет** |
| `src/infra/database.*` | **Да** — подключение через `libpq` вместо `sqlite3` |
| `src/repositories/*` | **Да, частично** — диалект SQL и плейсхолдеры (`?` → `$1`) |
| `db/schema.sql` | **Да** — типы (`INTEGER PRIMARY KEY AUTOINCREMENT` → `SERIAL/BIGSERIAL`), `datetime('now')` → `now()` |

Конкретные правки в репозиториях минимальны:

- **Плейсхолдеры параметров.** В SQLite — `?`, в PostgreSQL — `$1, $2, ...`. Логика prepared
  statement (шаг `07`) та же — меняется только синтаксис плейсхолдера.
- **Автоинкремент и id.** `INTEGER PRIMARY KEY AUTOINCREMENT` → `BIGSERIAL PRIMARY KEY`;
  получение нового id — через `RETURNING id` вместо `last_insert_rowid()`.
- **Даты.** `datetime('now')` → `now()`; функции `date(...)` похожи, но не идентичны.
- **Транзакции, `JOIN`, `GROUP BY`, `WITH`** (наши отчёты шага `13`) — **работают почти
  без изменений**, это стандартный SQL.

> **Главная мысль шага в одну фразу.** Благодаря слоям нам **не придётся трогать ни одного
> бизнес-правила** — только переписать «как разговаривать с базой» в `infra/database` и чуть
> поправить SQL в репозиториях. Это и есть практическая выгода архитектуры из шага `03`:
> заказчик может потребовать PostgreSQL, и это будет **точечная работа**, а не переписывание
> системы.

## 6. TLS через обратный прокси nginx (шифрование передачи, ТЗ п. 3.4)

ТЗ (п. 3.4, ГОСТ/ISO 17799) требует **защиту передачи данных**. Наш cpp-httplib отдаёт
обычный HTTP. Встраивать TLS прямо в C++ — лишняя возня (сертификаты, OpenSSL в коде).
Промышленный приём: поставить перед нашим сервером **обратный прокси** (reverse proxy)
**nginx**, который:

1. принимает **зашифрованные** HTTPS-запросы снаружи (порт `443`, сертификат TLS);
2. расшифровывает их и передаёт нашему серверу по локальному HTTP (`127.0.0.1:8080`);
3. ответ шифрует обратно и отдаёт клиенту.

```
[ Клиент ] --HTTPS (443, зашифровано)--> [ nginx ] --HTTP (127.0.0.1:8080)--> [ flowershop.exe ]
```

> **Аналогия.** nginx — это **охранник на входе с защищённым каналом связи**. Снаружи с ним
> говорят только по зашифрованной линии. Внутри здания (на одном сервере, `127.0.0.1`) он
> передаёт сообщение нашему серверу открыто — но это уже **внутри периметра**, безопасно.

Минимальный фрагмент конфига nginx:

```nginx
server {
    listen 443 ssl;
    server_name flowershop.local;

    ssl_certificate     /etc/nginx/certs/fullchain.pem;   # сертификат TLS
    ssl_certificate_key /etc/nginx/certs/privkey.pem;      # приватный ключ

    location / {
        proxy_pass http://127.0.0.1:8080;   # наш C++-сервер, доступен ТОЛЬКО локально
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

> При этом порт `8080` нашего сервера **закрываем** в брандмауэре наружу (раздел 3.4) и
> привязываем сервер к `127.0.0.1` — чтобы напрямую, в обход TLS, до него было не достучаться.
> Так выполняется «защита передачи» из ТЗ, **не усложняя наш C++-код**.
