# Шаг 14. Тестирование на doctest

> **Цель шага:** перестать проверять код «на глаз» и научить его проверять себя **сам**. В
> конце у вас будет: понимание, зачем нужны тесты; подключённая в CMake библиотека **doctest**
> и отдельная цель сборки `tests`; первые юнит-тесты на `PricingService` и на защиту от дубля
> в `OrderService` — с **тестовым репозиторием на базе в памяти** (`sqlite ":memory:"`); а
> также набор команд `curl` для **ручной проверки API**. Это закрывает раздел 4 ТЗ (порядок
> контроля и приёмки) и даёт основу для документа «Программа и методика испытаний» (раздел 5).

> Имена классов и таблиц — строго из предыдущих шагов (`PricingService`, `OrderService`,
> `OrderRepository`, `Database`, `orders`, `order_items`, `products`, ...). Стек тестов —
> **doctest** (header-only, один файл `doctest.h`), как зафиксировано в спецификации.

## 1. Зачем вообще тесты (и почему «у меня и так работает» — ловушка)

Вы написали `PricingService` и проверили руками: запустили сервер, оформили заказ, цена
посчиталась — отлично. Через неделю вы меняете расчёт цены (например, добавляете скидку).
Запускаете — вроде работает. Но **не сломалось ли что-то ещё**, что вы трогали месяц назад
и уже забыли? Проверять руками **весь** проект после каждого изменения невозможно.

**Тест** — это маленькая программа, которая вызывает ваш код с известными данными и
проверяет, что результат именно такой, как ожидалось. Запускается за секунду, и **сразу
краснеет**, если что-то сломалось.

> **Бытовая аналогия.** Тесты — это **сигнализация в доме**. Вы не обходите все окна
> вручную каждый вечер. Вы один раз поставили датчики, и теперь, если кто-то полез в окно,
> сирена сразу орёт. Тест — датчик на конкретное окно («цена букета должна быть 2500»). Чем
> больше датчиков, тем спокойнее менять код.

Главное слово этого шага — **регрессия**. Регрессия — это когда **раньше работавшее
сломалось** после ваших изменений. Тесты ловят регрессии автоматически: вы поменяли код,
запустили тесты, один покраснел — значит, вы что-то сломали, чините **сразу**, пока помните,
что трогали. Без тестов регрессия всплывёт у заказчика через месяц, и искать её будете долго.

Виды тестов, которые нам важны:

- **Юнит-тест** (unit = «единица») — проверяет **одну маленькую деталь** в изоляции: один
  метод сервиса. Быстрый, не трогает сеть и (в идеале) настоящую БД.
- **Ручное тестирование API** — мы сами шлём запросы через `curl`/Postman и смотрим ответы.
  Медленнее, но проверяет систему «целиком», как её увидит клиент.

## 2. Как устроен doctest и зачем header-only

**doctest** — это библиотека тестирования, состоящая из **одного заголовочного файла**
`doctest.h`. Никакой установки: кладём файл в `third_party/` (как `httplib.h`, `json.hpp`) и
подключаем `#include`. «Header-only» означает «весь код библиотеки — в заголовке, отдельную
библиотеку линковать не надо» (про линковку — шаг `01`/`05`).

Тест на doctest выглядит так:

```cpp
TEST_CASE("человеко-читаемое название теста") {
    // ... вызываем код ...
    CHECK(результат == ожидание);   // если неверно — тест краснеет, но идёт дальше
    REQUIRE(указатель != nullptr);  // если неверно — тест краснеет и СРАЗУ останавливается
}
```

- `TEST_CASE("...")` — один тест. Имя пишем по-человечески: что проверяем.
- `CHECK(условие)` — проверка. Не сбылась → тест провален, но остальные `CHECK` в этом
  тесте ещё выполнятся (увидим все ошибки сразу).
- `REQUIRE(условие)` — то же, но если не сбылось, дальше идти **нельзя** (например, объект
  не создался — нет смысла проверять его поля). Тест немедленно прерывается.
- `CHECK_THROWS_AS(выражение, ТипИсключения)` — проверка, что код **бросил** именно такое
  исключение (нам это нужно для дубля заказа: `DuplicateOrderError`).

## 3. Подключаем doctest в CMake (отдельная цель `tests`)

У нас уже есть `CMakeLists.txt` из шага `05`, который собирает сервер (цель, например,
`flowershop`). Тесты — это **отдельная программа** (свой `main`), поэтому делаем для них
**отдельную цель** (target) `tests`. Так сервер и тесты собираются независимо: тесты не
попадут в релизный бинарник.

> **Что такое «цель» (target) в CMake.** Это «что мы собираем»: один исполняемый файл или
> одна библиотека. У проекта может быть несколько целей. У нас две: `flowershop` (сервер) и
> `tests` (тесты).

Добавьте в конец `CMakeLists.txt`:

```cmake
# ---------- Цель для тестов ----------
# Список исходников проекта, которые тестируем (БЕЗ main.cpp сервера — у тестов свой main!).
# Удобно вынести общие .cpp в переменную, чтобы и сервер, и тесты брали один список.
set(APP_SOURCES
    src/infra/database.cpp
    src/repositories/order_repository.cpp
    src/repositories/product_repository.cpp
    src/services/pricing_service.cpp
    src/services/order_service.cpp
    # ... остальные .cpp проекта (кроме src/main.cpp) ...
)

# Файлы самих тестов:
set(TEST_SOURCES
    tests/test_main.cpp
    tests/test_pricing_service.cpp
    tests/test_order_service.cpp
)

add_executable(tests ${TEST_SOURCES} ${APP_SOURCES})

# Тестам нужны те же заголовки (third_party) и тот же стандарт C++17.
target_include_directories(tests PRIVATE third_party src)
target_compile_features(tests PRIVATE cxx_std_17)

# sqlite3 компилируем как часть проекта (как и для сервера, шаг 05):
target_sources(tests PRIVATE third_party/sqlite3.c)

# Регистрируем цель в системе тестов CMake (ctest), чтобы запускать одной командой:
enable_testing()
add_test(NAME unit_tests COMMAND tests)
```

Сборка и запуск тестов:

```bash
# Из корня проекта flowershop/
cmake -S . -B build                 # настроить (создаст папку build)
cmake --build build --target tests  # собрать ТОЛЬКО тесты
./build/tests                       # запустить тесты напрямую (увидим отчёт doctest)

# Либо через ctest (он сам найдёт наши тесты):
ctest --test-dir build --output-on-failure
```

Если всё хорошо — увидите что-то вроде `[doctest] test cases: 3 | 3 passed`. Если тест
упал — doctest покажет, **в каком файле, в какой строке** и **что ожидалось против что
получилось**.

## 4. Файл `tests/test_main.cpp` — точка входа тестов

doctest сам сгенерирует функцию `main` для тестовой программы, если мы попросим его об этом
**ровно в одном** `.cpp` специальной строкой-определением. Этот файл больше ничего не
содержит — только «включатель».

```cpp
// tests/test_main.cpp
// Эта строка ДО #include заставляет doctest сгенерировать функцию main().
// Она должна быть РОВНО в ОДНОМ файле проекта, иначе будет ошибка линковки
// (две функции main). Остальные тестовые файлы просто включают doctest.h.
#define DOCTEST_CONFIG_IMPLEMENT_WITH_MAIN
#include "doctest.h"
```

Во всех **остальных** тестовых файлах подключаем doctest **без** этого определения:

```cpp
#include "doctest.h"   // в test_pricing_service.cpp, test_order_service.cpp и т.д.
```

## 5. Тестовый репозиторий на базе в памяти (`":memory:"`)

Юнит-тест должен быть **быстрым и чистым**: он не должен портить настоящий файл
`flowershop.db` и зависеть от того, что в нём накопилось. Решение красивое: SQLite умеет
держать **всю базу в оперативной памяти**, если открыть её по специальному «имени файла»
`":memory:"`. Такая база:

- создаётся пустой при открытии;
- живёт, пока открыто подключение;
- **исчезает** при закрытии — ничего на диске не остаётся.

То есть для каждого теста мы поднимаем **свежую, идеально чистую базу**, заливаем туда схему
(`schema.sql`) и нужные данные — и работаем. Это и есть «поддельный/тестовый репозиторий»: на
самом деле репозиторий **настоящий**, просто база у него временная, в памяти.

> **Почему не «мок» (mock-объект)?** В больших проектах вместо БД часто подставляют
> «поддельный» объект-заглушку. Но в C++ для этого пришлось бы заводить интерфейсы
> (абстрактные классы) у каждого репозитория — лишняя сложность для новичка. База в памяти
> даёт тот же эффект (быстро, изолированно, реальный SQL проверяется по-настоящему) **без**
> переусложнения. Это сознательный учебный выбор.

Сделаем маленький помощник, который поднимает чистую базу в памяти и заливает схему:

```cpp
// tests/test_helpers.h
#pragma once
#include <fstream>
#include <sstream>
#include <string>
#include "../src/infra/database.h"   // fs::Database (шаг 07)

namespace fs {

// Прочитать весь файл в строку (нужно, чтобы залить db/schema.sql в базу).
inline std::string readFile(const std::string& path) {
    std::ifstream in(path);
    std::stringstream ss;
    ss << in.rdbuf();
    return ss.str();
}

// Поднять ЧИСТУЮ базу в памяти и создать в ней все таблицы из schema.sql.
inline Database makeMemoryDb() {
    Database db(":memory:");                 // ":memory:" = база живёт в ОЗУ, не на диске
    db.execScript(readFile("db/schema.sql"));// выполнить весь SQL схемы (CREATE TABLE ...)
    db.execScript(readFile("db/seed.sql"));  // справочники: роли, статусы, категории
    return db;
}

} // namespace fs
```

> `execScript` — это метод нашей обёртки `Database` (шаг `07`), который выполняет сразу
> много SQL-команд из строки (а не одну). Если в вашем коде он называется иначе —
> поправьте; в `sqlite3` за это отвечает `sqlite3_exec`.
> Путь `db/schema.sql` подразумевает запуск тестов из корня проекта. Если запускаете из
> `build/`, передайте абсолютный путь или скопируйте `db/` в `build/` (см. примечание в
> конце раздела 7).

## 6. `tests/test_pricing_service.cpp` — первый юнит-тест

`PricingService::priceOf` (шаг `09`) очень прост: цена единицы товара = `products.base_price`.
Проверим: добавим товар с известной ценой и убедимся, что сервис вернул именно её. А ещё —
что для несуществующего товара он бросает `NotFoundError`.

```cpp
// tests/test_pricing_service.cpp
#include "doctest.h"
#include "test_helpers.h"
#include "../src/repositories/product_repository.h"
#include "../src/services/pricing_service.h"
#include "../src/services/service_errors.h"   // NotFoundError (шаг 08)

TEST_CASE("PricingService возвращает base_price товара") {
    // (1) Чистая база в памяти со схемой.
    fs::Database db = fs::makeMemoryDb();

    // (2) Кладём товар с известной ценой 2500.0 напрямую через репозиторий.
    fs::ProductRepository products(db);
    fs::Product p;
    p.name       = "Букет 'Нежность'";
    p.kind       = "bouquet";       // допустимое значение CHECK из схемы (шаг 04)
    p.base_price = 2500.0;
    p.is_active  = true;
    int64_t id = products.save(p);  // INSERT INTO products, вернёт новый id

    // (3) Сервис, которому выдали репозиторий (инъекция зависимости, шаг 09).
    fs::PricingService pricing(products);

    // (4) Проверяем: цена единицы = base_price.
    CHECK(pricing.priceOf(id) == doctest::Approx(2500.0));
    //                              ^^^^^^^^^^^^^ сравнение double через Approx:
    //   нельзя сравнивать дробные числа на точное ==, могут быть микро-погрешности.
}

TEST_CASE("PricingService бросает NotFoundError для несуществующего товара") {
    fs::Database db = fs::makeMemoryDb();
    fs::ProductRepository products(db);
    fs::PricingService pricing(products);

    // Товара с id=999 нет -> ожидаем исключение именно типа NotFoundError.
    CHECK_THROWS_AS(pricing.priceOf(999), fs::NotFoundError);
}
```

Разбор для новичка:

- `doctest::Approx(2500.0)` — «примерно 2500». Так сравнивают `double` (см. оговорку про
  деньги в шаге `09`): прямое `== 2500.0` иногда подводит из-за погрешностей плавающей точки.
- `CHECK_THROWS_AS(..., fs::NotFoundError)` — тест **проходит**, только если код бросил
  именно это исключение. Если бы `priceOf` молча вернул 0 — тест бы покраснел, и правильно:
  «молча вернуть 0 за несуществующий товар» — это баг.

## 7. `tests/test_order_service.cpp` — проверяем защиту от дубля

Это тест поинтереснее: проверим **бизнес-правило** из шага `09` — `OrderService::createOrder`
должен бросить `DuplicateOrderError`, если тот же клиент за короткое время оформляет
**такой же** заказ второй раз. Это ключевое требование ТЗ (раздел 2: «дубли заказов»).

Сценарий теста:

1. Поднять чистую базу, положить клиента, продавца, товар с остатком на складе.
2. Собрать `OrderService` со всеми его зависимостями (как «сборочный цех» из шага `09`).
3. Оформить заказ — ожидаем успех.
4. Тут же оформить **идентичный** заказ — ожидаем `DuplicateOrderError`.

```cpp
// tests/test_order_service.cpp
#include "doctest.h"
#include "test_helpers.h"
#include "../src/repositories/order_repository.h"
#include "../src/repositories/product_repository.h"
#include "../src/repositories/stock_repository.h"
#include "../src/repositories/write_off_repository.h"
#include "../src/repositories/notification_repository.h"
#include "../src/repositories/user_repository.h"
#include "../src/services/pricing_service.h"
#include "../src/services/stock_service.h"
#include "../src/services/notification_service.h"
#include "../src/services/order_service.h"
#include "../src/services/service_errors.h"

// Маленький помощник: создать товар с остатком на складе и вернуть его product_id.
static int64_t seedProductWithStock(fs::Database& db, double price, int qty) {
    fs::ProductRepository products(db);
    fs::Product p;
    p.name = "Роза красная"; p.kind = "flower"; p.base_price = price; p.is_active = true;
    int64_t pid = products.save(p);

    fs::StockRepository stock(db);
    fs::Stock s;
    s.product_id = pid; s.quantity = qty; s.quality = "отличное";  // CHECK из схемы шага 04
    stock.save(s);
    return pid;
}

TEST_CASE("OrderService запрещает дубль заказа (требование ТЗ)") {
    fs::Database db = fs::makeMemoryDb();

    // (1) Данные: клиент, товар с остатком.
    fs::UserRepository users(db);
    int64_t clientId = users.createClient("client1", "Иван Клиентов"); // упрощённый helper
    int64_t productId = seedProductWithStock(db, /*price*/100.0, /*qty*/50);

    // (2) Собираем сервис со всеми зависимостями (сборочный цех из шага 09).
    fs::OrderRepository        orderRepo(db);
    fs::StockRepository        stockRepo(db);
    fs::WriteOffRepository     writeOffRepo(db);
    fs::ProductRepository      productRepo(db);
    fs::NotificationRepository notifRepo(db);

    fs::NotificationService notifSvc(notifRepo);
    fs::StockService        stockSvc(stockRepo, writeOffRepo, notifSvc);
    fs::PricingService      pricingSvc(productRepo);
    fs::OrderService        orderSvc(orderRepo, stockSvc, pricingSvc, notifSvc);

    // (3) Готовим заказ: 3 розы от нашего клиента.
    fs::Order order;
    order.client_id = clientId;
    fs::OrderItem item;
    item.product_id = productId;
    item.quantity   = 3;
    order.items.push_back(item);

    // Первый заказ оформляется успешно: id присвоен, цена посчитана сервисом.
    fs::Order created = orderSvc.createOrder(order);
    REQUIRE(created.id > 0);                              // заказ реально сохранён
    CHECK(created.total_price == doctest::Approx(300.0)); // 3 * 100.0, цену считает сервис

    // (4) Тут же — идентичный заказ. Ожидаем DuplicateOrderError (-> 409 в API).
    CHECK_THROWS_AS(orderSvc.createOrder(order), fs::DuplicateOrderError);
}

TEST_CASE("OrderService отвергает заказ без позиций (валидация)") {
    fs::Database db = fs::makeMemoryDb();
    fs::OrderRepository        orderRepo(db);
    fs::StockRepository        stockRepo(db);
    fs::WriteOffRepository     writeOffRepo(db);
    fs::ProductRepository      productRepo(db);
    fs::NotificationRepository notifRepo(db);
    fs::NotificationService notifSvc(notifRepo);
    fs::StockService        stockSvc(stockRepo, writeOffRepo, notifSvc);
    fs::PricingService      pricingSvc(productRepo);
    fs::OrderService        orderSvc(orderRepo, stockSvc, pricingSvc, notifSvc);

    fs::Order empty;             // ни одной позиции
    CHECK_THROWS_AS(orderSvc.createOrder(empty), fs::ValidationError);
}
```

> Обратите внимание: тест почти **дословно повторяет «сборочный цех»** из раздела 8 шага `09`.
> Это не случайно — именно поэтому инъекция зависимостей (передача репозиториев в
> конструктор) так удобна: в тесте мы собираем сервис из тех же кубиков, но с базой в памяти.
> Если бы `OrderService` сам создавал репозитории внутри себя с файлом `flowershop.db`, мы бы
> **не смогли** подсунуть ему тестовую базу — пришлось бы портить настоящую.

> **Замечание про методы-помощники.** `users.createClient(...)`, `productRepo.save(...)`,
> `stockRepo.save(...)` — это методы репозиториев из шага `07`. Если у вас они называются
> иначе (например, заполнение через прямой `INSERT` в `db.execScript`) — суть та же: положить
> в чистую базу минимум данных, нужных тесту. Запускать тесты лучше из корня проекта, чтобы
> относительный путь `db/schema.sql` нашёлся (или скопируйте папку `db/` в `build/`).

## 8. Ручное тестирование API через curl / Postman

Юнит-тесты проверяют детали. Но заказчик увидит систему **снаружи**, через HTTP. Поэтому
перед сдачей мы прогоняем **сквозной сценарий** руками — это и есть основа «Программы и
методики испытаний» (раздел 5 ТЗ). Удобнее всего через `curl` (консоль) или **Postman**
(программа с окошками).

Сквозной сценарий «вход → создание заказа → проверка»:

```bash
# Шаг 1. Войти и получить токен (аутентификация, шаг 10).
curl -X POST http://localhost:8080/api/auth/login \
     -H "Content-Type: application/json" \
     -d '{"login":"seller1","password":"секрет"}'
# Ожидаем 200 и тело вида: {"token":"...."}
# Скопируйте токен в переменную окружения, чтобы не вставлять каждый раз:
TOKEN="сюда_вставьте_токен_из_ответа"

# Шаг 2. Посмотреть каталог товаров (публичный GET).
curl http://localhost:8080/api/products
# Ожидаем 200 и JSON-массив товаров.

# Шаг 3. Создать заказ (POST + токен + JSON-тело).
curl -X POST http://localhost:8080/api/orders \
     -H "Authorization: Bearer $TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"client_id": 6, "items": [{"product_id": 1, "quantity": 3}]}'
# Ожидаем 201 Created и JSON созданного заказа (с посчитанной total_price).

# Шаг 4. Повторить ТОТ ЖЕ запрос ещё раз (проверяем защиту от дубля!).
curl -X POST http://localhost:8080/api/orders \
     -H "Authorization: Bearer $TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"client_id": 6, "items": [{"product_id": 1, "quantity": 3}]}'
# Ожидаем 409 Conflict — система не дала создать дубль (требование ТЗ выполнено!).

# Шаг 5. Запрос без токена — проверяем защиту.
curl -i -X POST http://localhost:8080/api/orders \
     -H "Content-Type: application/json" \
     -d '{"client_id": 6, "items": [{"product_id": 1, "quantity": 3}]}'
# Ожидаем 401 Unauthorized.
```

> Флаг `-i` показывает заголовки ответа вместе с **кодом состояния** (`HTTP/1.1 409 Conflict`).
> Для ручной проверки коды (200/201/400/401/403/404/409) — самое важное: они и есть «приёмочный
> критерий» для каждого сценария.

Эти же сценарии в **Postman** делаются мышкой: создаёте запрос, выбираете метод
(`POST`), вписываете URL, на вкладке *Headers* добавляете `Authorization: Bearer ...`, на
вкладке *Body → raw → JSON* — тело. Postman удобно **сохраняет** набор запросов в коллекцию,
которую можно отдать вместе с проектом как «методику испытаний».

**Таблица-чеклист ручных проверок** (готовая заготовка для «Программы и методики испытаний»):

| № | Что проверяем | Запрос | Ожидаемый код | Ожидаемый результат |
|---|---------------|--------|---------------|---------------------|
| 1 | Вход с верным паролем | `POST /api/auth/login` | 200 | в теле есть `token` |
| 2 | Вход с неверным паролем | `POST /api/auth/login` | 401 | отказ |
| 3 | Список товаров | `GET /api/products` | 200 | массив товаров |
| 4 | Создание заказа | `POST /api/orders` | 201 | заказ с `total_price` |
| 5 | Дубль заказа | повтор п.4 | 409 | дубль отклонён |
| 6 | Заказ без токена | `POST /api/orders` без заголовка | 401 | отказ |
| 7 | Отчёт прибыли курьером | `GET /api/reports/profit` токеном курьера | 403 | нет права `report.view` |
| 8 | Отчёт прибыли владельцем | `GET /api/reports/profit` токеном владельца | 200 | числа отчёта |
