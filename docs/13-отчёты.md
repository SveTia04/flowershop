# Шаг 13. Отчёты — аналитика для владельца (`ReportService`)

> **Цель шага:** научить систему отвечать на вопросы владельца магазина: «сколько мы
> продали за месяц?», «какая была прибыль?», «какие товары берут чаще всего?», «как нас
> оценивают клиенты?». В конце вы напишете сервис `ReportService` с SQL-запросами-отчётами,
> структуры-результаты под них, контроллер `ReportController` с эндпоинтами
> `GET /api/reports/sales`, `/api/reports/profit`, `/api/reports/top-products` — доступными
> **только** ролям `owner` и `admin` (право `report.view`). А ещё реализуете упрощённое
> «предупреждение о спросе» — список товаров, которые **популярны, но их мало на складе**.

> Диаграммы — на языке **Mermaid**, рядом ASCII-дубликат. Имена таблиц и полей — строго из
> шага `04` (`orders`, `order_items`, `supply_items`, `reviews`, `products`, `stock` и их
> поля `total_price`, `price_each`, `quantity`, `purchase_price`, `rating`, `created_at`).
> Имена сервисов/контроллеров — из шагов `09` и `11` (`ReportService`, `AuthorizationService`,
> право `report.view`).

## 1. Зачем отдельный шаг про отчёты (и почему это требование ТЗ)

Раздел 3.3 ТЗ среди функций **Владельца** прямо называет: смотреть отчёты по продажам, по
прибыли, по популярным товарам, оценивать работу магазина. А раздел 2 ТЗ перечисляет
проблему: *«сложно делать отчёты вручную»* — бухгалтер или владелец сводит цифры из тетрадок
и чеков, тратит часы и ошибается. Наша система должна выдавать эти цифры **за секунду по
кнопке**.

> **Бытовая аналогия.** Представьте, что в конце месяца вы вручную перебираете коробку с
> чеками, складываете суммы на калькуляторе, отдельно считаете, что почём купили у
> поставщика, вычитаете — и так по каждому товару. Отчёт — это когда вместо коробки с чеками
> у вас есть **умная таблица**, которая сама всё сложит и вычтет, как только вы спросите.

Ключевая идея шага одной фразой: **отчёт — это не новые данные, а свежий взгляд на уже
имеющиеся**. Мы ничего не добавляем в базу; мы только **читаем и считаем** по тем строкам,
что уже накопились в `orders`, `order_items`, `supply_items`, `reviews`.

## 2. Главная мысль: отчёт = SQL-агрегация

До этого шага наши SQL-запросы доставали **отдельные строки** («дай заказ с id=42»).
Отчёт устроен иначе: он **сворачивает много строк в одно число** (или в несколько чисел по
группам). Это называется **агрегация**.

Четыре «волшебных слова» SQL для агрегации:

| Функция / слово | Что делает | Бытовой смысл |
|-----------------|-----------|----------------|
| `SUM(x)` | складывает все значения столбца `x` | «итого выручка» |
| `COUNT(*)` | считает количество строк | «сколько заказов» |
| `AVG(x)` | среднее значение | «средний чек», «средняя оценка» |
| `GROUP BY y` | разбивает строки на группы по `y` и считает агрегаты **внутри каждой группы** | «продажи **по каждому товару** отдельно» |

> **Аналогия для `GROUP BY`.** Представьте стопку чеков. `SUM` без `GROUP BY` — сложить
> **всю** стопку в одну сумму. `SUM ... GROUP BY product_id` — сначала **разложить чеки по
> кучкам** (одна кучка = один товар), а потом сложить каждую кучку отдельно. На выходе — не
> одно число, а табличка «товар → сумма».

Ещё два слова, без которых отчёты не живут:

- `WHERE` — **фильтр строк ДО** группировки («только заказы за этот месяц»).
- `ORDER BY ... LIMIT N` — **сортировка и обрезка** («самые популярные — первые 10»).

> **Важно про даты в SQLite.** Все даты у нас хранятся как текст в формате `'2026-06-19 ...'`
> (см. `DEFAULT (datetime('now'))` в схеме шага `04`). Поэтому период «с… по…» — это
> обычное сравнение строк: `created_at >= '2026-06-01' AND created_at < '2026-07-01'`.
> Текстовые даты в формате `ГГГГ-ММ-ДД` сравниваются как обычные строки и при этом дают
> правильный хронологический порядок — это удобное свойство такого формата.

## 3. Четыре отчёта проекта (сначала SQL, потом код)

Разберём четыре запроса. Каждый сначала объясним «на пальцах», потом дадим SQL с
комментариями, потом покажем, какой результат он возвращает.

### 3.1. Отчёт продаж за период

**Вопрос владельца:** «На сколько денег мы напродавали с 1 по 30 июня и сколько было
заказов?»

Считаем **только завершённые продажи** — заказы в статусе `done` (выполнен). Отменённые
(`cancelled`) и возвраты (`returned`) в выручку не идут. Коды статусов — из `seed.sql`
шага `04`; их числовые `id` берём из справочника `order_statuses` подзапросом, чтобы не
зашивать «магическое число» в код.

```sql
-- Отчёт продаж за период [:from, :to).
-- Берём только заказы со статусом 'done' (выполнен) — это реальная выручка.
SELECT
    COUNT(*)            AS orders_count,     -- сколько было выполненных заказов
    COALESCE(SUM(o.total_price), 0) AS revenue   -- суммарная выручка (0, если заказов нет)
FROM orders AS o
WHERE o.status_id = (SELECT id FROM order_statuses WHERE code = 'done')
  AND o.created_at >= :from      -- начало периода включительно, напр. '2026-06-01'
  AND o.created_at <  :to;       -- конец периода НЕ включительно, напр. '2026-07-01'
```

Пояснения новичку:

- `:from` и `:to` — это **параметры** (плейсхолдеры). В коде мы подставим вместо них
  настоящие значения через **prepared statement** (шаг `07`), а не склейкой строк. Это
  защита от **SQL-инъекций**.
- `COALESCE(SUM(...), 0)` — если выполненных заказов нет, `SUM` вернёт `NULL` (пусто).
  `COALESCE(x, 0)` означает «возьми `x`, а если он `NULL` — возьми `0»`. Так отчёт всегда
  отдаёт число, а не «пусто».
- `AS revenue` — это **псевдоним** столбца: имя, под которым результат вернётся в код.

Результат — **одна строка из двух чисел**: например, `orders_count = 87`, `revenue = 215300.0`.

> Если владельцу нужна разбивка **по дням**, добавляется группировка по дате:
> `... GROUP BY date(o.created_at) ORDER BY date(o.created_at)`. Тогда вернётся уже
> **много строк** — по одной на каждый день. Это естественное расширение того же отчёта.

### 3.2. Отчёт прибыли

**Вопрос владельца:** «Прибыль — это не вся выручка. Сколько мы заработали **сверх** того,
что потратили на закупку у поставщиков?»

Прибыль (упрощённо, учебно) считаем так:

```
прибыль = выручка по проданным позициям − закупочная стоимость этих товаров
```

- **Выручка по позициям** — это `SUM(order_items.price_each * order_items.quantity)` по
  выполненным заказам.
- **Закупочная стоимость** — сколько **в среднем** стоила у поставщика единица товара. Эту
  цену берём из `supply_items.purchase_price` (см. шаг `04` — поле закупочной цены в
  позиции поступления). Один и тот же товар мог поступать партиями по разной цене, поэтому
  берём **среднюю закупочную цену** `AVG(purchase_price)` по товару.

> **Честная оговорка (упрощение).** «Правильная» бухгалтерская себестоимость учитывает,
> из какой именно партии ушёл проданный цветок (FIFO/FEFO, см. `StockService` в шаге `09`).
> Точный учёт себестоимости — отдельная большая тема; для учебного отчёта мы берём среднюю
> закупочную цену по товару. Это явно отмечено и в карте требований ТЗ (приложение).

```sql
-- Отчёт прибыли за период [:from, :to).
-- Сначала во вложенном запросе считаем среднюю закупочную цену по каждому товару.
WITH avg_cost AS (                       -- WITH = временная «таблица на один запрос» (CTE)
    SELECT product_id,
           AVG(purchase_price) AS unit_cost   -- средняя закупочная цена единицы товара
    FROM supply_items
    GROUP BY product_id
)
SELECT
    COALESCE(SUM(oi.price_each * oi.quantity), 0)          AS revenue,      -- выручка
    COALESCE(SUM(ac.unit_cost  * oi.quantity), 0)          AS cost,         -- себестоимость
    COALESCE(SUM(oi.price_each * oi.quantity), 0)
      - COALESCE(SUM(ac.unit_cost * oi.quantity), 0)       AS profit        -- прибыль
FROM order_items AS oi
JOIN orders AS o      ON o.id = oi.order_id          -- к позиции цепляем её заказ
LEFT JOIN avg_cost ac ON ac.product_id = oi.product_id  -- и среднюю закупку этого товара
WHERE o.status_id = (SELECT id FROM order_statuses WHERE code = 'done')
  AND o.created_at >= :from
  AND o.created_at <  :to;
```

Что тут нового для новичка:

- `WITH avg_cost AS (...)` — это **CTE** (Common Table Expression), «временная таблица,
  живущая только внутри этого запроса». Мы посчитали среднюю закупку отдельно, дали ей имя
  `avg_cost`, и дальше пользуемся как обычной таблицей. Это просто способ не писать длинный
  подзапрос прямо посреди основного запроса — чтобы было читаемо.
- `JOIN` — **соединение таблиц**: «к каждой позиции `order_items` приклей данные её заказа
  `orders` (по `order_id`) и закупочную цену (по `product_id`)». Без `JOIN` мы бы видели
  только `product_id`, но не его закупочную цену.
- `LEFT JOIN` (а не просто `JOIN`) для `avg_cost` — на случай, если у товара **вообще нет**
  записей о поступлении (его никогда не закупали официально). Тогда `unit_cost` будет
  `NULL`, и эта позиция просто не добавит себестоимости. Обычный `JOIN` молча выкинул бы
  такую позицию из выручки — а это исказило бы цифры.

Результат — **одна строка**: `revenue`, `cost`, `profit`.

### 3.3. Топ популярных товаров

**Вопрос владельца:** «Что у нас разбирают лучше всего? На что делать ставку при закупке?»

«Популярность» = сколько **штук** товара продано (сумма `quantity` по позициям выполненных
заказов). Группируем по товару, сортируем по убыванию, берём первые `N`.

```sql
-- Топ-N популярных товаров за период [:from, :to).
SELECT
    p.id                       AS product_id,
    p.name                     AS product_name,
    SUM(oi.quantity)           AS sold_qty,     -- сколько штук продано
    SUM(oi.price_each * oi.quantity) AS revenue -- сколько денег принёс товар
FROM order_items AS oi
JOIN orders   AS o ON o.id = oi.order_id
JOIN products AS p ON p.id = oi.product_id      -- чтобы получить имя товара, а не голый id
WHERE o.status_id = (SELECT id FROM order_statuses WHERE code = 'done')
  AND o.created_at >= :from
  AND o.created_at <  :to
GROUP BY p.id, p.name          -- разбиваем строки по товарам (одна кучка = один товар)
ORDER BY sold_qty DESC         -- самые продаваемые — наверху
LIMIT :limit;                  -- берём только первые N (например, 10)
```

Результат — **много строк** (до `limit`), отсортированных от самого продаваемого: каждая
строка = товар + сколько штук + сколько денег.

### 3.4. Средняя оценка (рейтинг) по отзывам

**Вопрос владельца:** «Как нас оценивают? Доволен ли клиент?»

Оценки лежат в таблице `reviews` (поле `rating` — целое 1–5, см. шаг `04`). Считаем среднее
и количество отзывов.

```sql
-- Средняя оценка магазина (по всем отзывам) и сколько всего отзывов.
SELECT
    COALESCE(AVG(rating), 0) AS avg_rating,    -- средняя оценка (0, если отзывов нет)
    COUNT(*)                 AS reviews_count  -- сколько отзывов всего
FROM reviews;
```

Результат — **одна строка**: `avg_rating` (например, `4.6`) и `reviews_count` (например,
`53`). При желании можно профильтровать по периоду (`WHERE created_at >= :from ...`) —
поле `created_at` в `reviews` для этого есть.

## 4. Структуры-результаты отчётов (домен отчётов)

Каждый SQL-отчёт возвращает строки определённой формы. В C++ мы описываем эту форму
**структурами** (`struct`) — так же, как доменные сущности в шаге `06`, только это не
«вещи из жизни», а «результаты подсчёта». Кладём их в `src/services/report_service.h`
рядом с самим сервисом (отдельной доменной таблицы под них нет — это не данные БД, а
производные числа).

```cpp
// src/services/report_service.h
#pragma once
#include <cstdint>     // int64_t
#include <string>
#include <vector>
#include "../infra/database.h"   // Database (шаг 07)

namespace fs {

// Результат отчёта продаж за период (раздел 3.1) — одна строка из двух чисел.
struct SalesReport {
    std::string from;          // начало периода, как пришло в запрос ('2026-06-01')
    std::string to;            // конец периода
    int64_t     orders_count = 0;  // сколько выполненных заказов
    double      revenue      = 0.0;// суммарная выручка (деньги — double, учебно)
};

// Результат отчёта прибыли (раздел 3.2).
struct ProfitReport {
    std::string from;
    std::string to;
    double revenue = 0.0;      // выручка по проданным позициям
    double cost    = 0.0;      // закупочная стоимость проданного
    double profit  = 0.0;      // прибыль = revenue - cost
};

// Одна строка топа товаров (раздел 3.3).
struct TopProductRow {
    int64_t     product_id   = 0;
    std::string product_name;
    int64_t     sold_qty     = 0;   // сколько штук продано
    double      revenue      = 0.0; // сколько денег принёс товар
};

// Результат отчёта по среднему рейтингу (раздел 3.4).
struct RatingReport {
    double  avg_rating    = 0.0;
    int64_t reviews_count = 0;
};

// Одна строка предупреждения о спросе (раздел 6): популярный, но кончается.
struct DemandWarningRow {
    int64_t     product_id   = 0;
    std::string product_name;
    int64_t     sold_qty     = 0;   // насколько популярен (продано штук за период)
    int64_t     in_stock     = 0;   // сколько осталось на складе суммарно
};

} // namespace fs
```

> Почему `= 0` / `= 0.0` прямо в полях? Это **инициализация по умолчанию**: даже если мы
> забудем заполнить поле, там будет 0, а не «мусор». Для отчётов это важно — пустой период
> обязан дать нули, а не случайные числа.

## 5. Сервис `ReportService`

`ReportService` живёт в слое **сервисов** (шаг `03`/`09`). Особенность: в отличие от
`OrderService`, у него **нет бизнес-правил и побочных эффектов** — он ничего не меняет,
только читает и считает. Поэтому он работает напрямую с `Database&` (выполняет SELECT-ы),
а не через отдельный репозиторий: отдельный «ReportRepository» завёл бы кучу почти
одинаковых методов-обёрток без пользы. (Если хотите строго соблюсти «SQL — только в
репозиториях» — можно вынести запросы в `ReportRepository`; для учебного проекта мы держим
их в сервисе и честно это отмечаем.)

```cpp
// src/services/report_service.h  (продолжение — сам класс)
namespace fs {

class ReportService {
public:
    // Сервису выдают подключение к БД (инъекция зависимости, шаг 09).
    explicit ReportService(Database& db) : db_(db) {}

    // Отчёт продаж за период [from, to). Даты — строки 'ГГГГ-ММ-ДД'.
    SalesReport sales(const std::string& from, const std::string& to);

    // Отчёт прибыли за тот же период.
    ProfitReport profit(const std::string& from, const std::string& to);

    // Топ-limit популярных товаров за период.
    std::vector<TopProductRow> topProducts(const std::string& from,
                                            const std::string& to,
                                            int limit);

    // Средняя оценка магазина по отзывам.
    RatingReport rating();

    // Упрощённое «прогнозирование спроса»: популярные товары, которых мало (раздел 6).
    std::vector<DemandWarningRow> demandWarnings(const std::string& from,
                                                 const std::string& to,
                                                 int64_t lowStockThreshold);

private:
    Database& db_;   // & = ссылка: один и тот же объект БД на всё приложение (шаг 07)
};

} // namespace fs
```

Реализация. Здесь видно, как prepared statement (шаг `07`) безопасно подставляет даты, и
как агрегатный SELECT превращается в нашу структуру.

```cpp
// src/services/report_service.cpp
#include "report_service.h"

namespace fs {

// ----- 3.1. Продажи за период -----
SalesReport ReportService::sales(const std::string& from, const std::string& to) {
    SalesReport r;
    r.from = from;
    r.to   = to;

    // Готовим запрос с параметрами ? (защита от SQL-инъекций, шаг 07).
    Statement st = db_.prepare(
        "SELECT COUNT(*), COALESCE(SUM(total_price), 0) "
        "FROM orders "
        "WHERE status_id = (SELECT id FROM order_statuses WHERE code = 'done') "
        "  AND created_at >= ? AND created_at < ?");
    st.bind(1, from);    // вместо первого ? подставится from
    st.bind(2, to);      // вместо второго ? — to

    if (st.step()) {                 // step() = «прочитать следующую строку результата»
        r.orders_count = st.columnInt64(0);   // COUNT(*) — нулевой столбец
        r.revenue      = st.columnDouble(1);   // SUM(total_price) — первый столбец
    }
    return r;
}

// ----- 3.2. Прибыль за период -----
ProfitReport ReportService::profit(const std::string& from, const std::string& to) {
    ProfitReport r;
    r.from = from;
    r.to   = to;

    Statement st = db_.prepare(
        "WITH avg_cost AS ("
        "    SELECT product_id, AVG(purchase_price) AS unit_cost "
        "    FROM supply_items GROUP BY product_id) "
        "SELECT "
        "  COALESCE(SUM(oi.price_each * oi.quantity), 0), "
        "  COALESCE(SUM(ac.unit_cost  * oi.quantity), 0) "
        "FROM order_items oi "
        "JOIN orders o      ON o.id = oi.order_id "
        "LEFT JOIN avg_cost ac ON ac.product_id = oi.product_id "
        "WHERE o.status_id = (SELECT id FROM order_statuses WHERE code = 'done') "
        "  AND o.created_at >= ? AND o.created_at < ?");
    st.bind(1, from);
    st.bind(2, to);

    if (st.step()) {
        r.revenue = st.columnDouble(0);
        r.cost    = st.columnDouble(1);
        r.profit  = r.revenue - r.cost;   // прибыль считаем уже в C++ — нагляднее
    }
    return r;
}

// ----- 3.3. Топ популярных товаров -----
std::vector<TopProductRow> ReportService::topProducts(const std::string& from,
                                                       const std::string& to,
                                                       int limit) {
    std::vector<TopProductRow> rows;   // vector = «растущий массив» (шаг 02)

    Statement st = db_.prepare(
        "SELECT p.id, p.name, SUM(oi.quantity), "
        "       SUM(oi.price_each * oi.quantity) "
        "FROM order_items oi "
        "JOIN orders   o ON o.id = oi.order_id "
        "JOIN products p ON p.id = oi.product_id "
        "WHERE o.status_id = (SELECT id FROM order_statuses WHERE code = 'done') "
        "  AND o.created_at >= ? AND o.created_at < ? "
        "GROUP BY p.id, p.name "
        "ORDER BY SUM(oi.quantity) DESC "
        "LIMIT ?");
    st.bind(1, from);
    st.bind(2, to);
    st.bind(3, static_cast<int64_t>(limit));

    while (st.step()) {                // строк может быть много — читаем в цикле
        TopProductRow row;
        row.product_id   = st.columnInt64(0);
        row.product_name = st.columnText(1);
        row.sold_qty     = st.columnInt64(2);
        row.revenue      = st.columnDouble(3);
        rows.push_back(row);           // добавляем строку в результат
    }
    return rows;
}

// ----- 3.4. Средняя оценка -----
RatingReport ReportService::rating() {
    RatingReport r;
    Statement st = db_.prepare(
        "SELECT COALESCE(AVG(rating), 0), COUNT(*) FROM reviews");
    if (st.step()) {
        r.avg_rating    = st.columnDouble(0);
        r.reviews_count = st.columnInt64(1);
    }
    return r;
}

} // namespace fs
```

> Имена методов `prepare`, `bind`, `step`, `columnInt64`, `columnDouble`, `columnText` —
> это тонкая обёртка над библиотекой `sqlite3`, которую мы ввели в шаге `07` (классы
> `Database` и `Statement`). Если в вашем коде обёртка называется иначе — поправьте вызовы,
> суть та же: подготовить запрос, привязать параметры, прочитать строки.

## 6. «Прогнозирование спроса» — упрощённо и честно

ТЗ (раздел 3.3, функции системы/владельца) упоминает идею **прогнозирования спроса**.
Настоящее прогнозирование — это статистика и машинное обучение (тренды, сезонность,
скользящие средние). Для учебного проекта это **слишком тяжело и не нужно**. Поэтому мы
честно **упрощаем** требование до полезного практического сигнала:

> **Правило-«светофор»:** предупреди о товарах, которые одновременно **популярны** (хорошо
> продавались за период) и **заканчиваются** на складе (остаток ниже порога). Это товары, по
> которым стоит срочно заказать поставку, иначе магазин потеряет продажи.

То есть мы не «предсказываем будущее», а **соединяем два факта из прошлого и настоящего**:
«брали часто» (`order_items`) + «осталось мало» (`stock`).

```mermaid
flowchart TD
    A[demandWarnings(from, to, threshold)] --> B[Топ продаж за период<br/>SUM order_items.quantity по товару]
    B --> C[Остаток на складе<br/>SUM stock.quantity по товару]
    C --> D{продано много<br/>И остаток < порога?}
    D -- да --> E[в список предупреждений:<br/>product, sold_qty, in_stock]
    D -- нет --> F[пропустить]
    E --> G[вернуть список -> владельцу/закупщику]
    F --> G
```

**ASCII-дубликат:**

```
для каждого товара:
    sold_qty = сколько продано за период (order_items по done-заказам)
    in_stock = сколько осталось (сумма stock.quantity)
    если sold_qty > 0 И in_stock < порог:
        -> добавить в предупреждения (популярный, но кончается)
вернуть список, отсортированный «самый ходовой — первым»
```

SQL соединяет продажи и остаток по товару:

```sql
-- Популярные, но заканчивающиеся товары: продано > 0 И остаток < :threshold.
WITH sold AS (
    SELECT oi.product_id, SUM(oi.quantity) AS sold_qty
    FROM order_items oi
    JOIN orders o ON o.id = oi.order_id
    WHERE o.status_id = (SELECT id FROM order_statuses WHERE code = 'done')
      AND o.created_at >= :from AND o.created_at < :to
    GROUP BY oi.product_id
),
in_stock AS (
    SELECT product_id, COALESCE(SUM(quantity), 0) AS qty
    FROM stock GROUP BY product_id
)
SELECT p.id, p.name, s.sold_qty, COALESCE(ist.qty, 0) AS in_stock
FROM sold s
JOIN products p       ON p.id = s.product_id
LEFT JOIN in_stock ist ON ist.product_id = s.product_id
WHERE COALESCE(ist.qty, 0) < :threshold   -- остаток ниже порога = «кончается»
ORDER BY s.sold_qty DESC;                 -- самые ходовые — первыми
```

```cpp
// src/services/report_service.cpp  (продолжение)
std::vector<DemandWarningRow> ReportService::demandWarnings(const std::string& from,
                                                            const std::string& to,
                                                            int64_t lowStockThreshold) {
    std::vector<DemandWarningRow> rows;
    Statement st = db_.prepare(
        "WITH sold AS ("
        "  SELECT oi.product_id, SUM(oi.quantity) AS sold_qty "
        "  FROM order_items oi JOIN orders o ON o.id = oi.order_id "
        "  WHERE o.status_id = (SELECT id FROM order_statuses WHERE code = 'done') "
        "    AND o.created_at >= ? AND o.created_at < ? "
        "  GROUP BY oi.product_id), "
        "in_stock AS ("
        "  SELECT product_id, COALESCE(SUM(quantity),0) AS qty "
        "  FROM stock GROUP BY product_id) "
        "SELECT p.id, p.name, s.sold_qty, COALESCE(ist.qty,0) "
        "FROM sold s "
        "JOIN products p        ON p.id = s.product_id "
        "LEFT JOIN in_stock ist ON ist.product_id = s.product_id "
        "WHERE COALESCE(ist.qty,0) < ? "
        "ORDER BY s.sold_qty DESC");
    st.bind(1, from);
    st.bind(2, to);
    st.bind(3, lowStockThreshold);

    while (st.step()) {
        DemandWarningRow row;
        row.product_id   = st.columnInt64(0);
        row.product_name = st.columnText(1);
        row.sold_qty     = st.columnInt64(2);
        row.in_stock     = st.columnInt64(3);
        rows.push_back(row);
    }
    return rows;
}
```

> Это **заглушка-упрощение** настоящего прогноза — и так оно записано в карте требований
> (приложение). Зато оно реально полезно: закупщик видит конкретный список «закажи это, пока
> не кончилось». Естественное развитие — учитывать скорость продаж (штук в день) и срок
> поставки; но это уже за рамками учебного проекта.

## 7. Контроллер `ReportController` и эндпоинты (только для owner/admin)

Теперь «открываем дверь наружу» (как в шаге `08`). Отчёты — чувствительные данные (прибыль!),
поэтому доступ строго по праву **`report.view`**, которое в шаге `11` выдано ролям `owner` и
`admin`. Проверяем его тем же middleware `requirePermission`, что и везде: сначала
`requireAuth` (есть ли валидный токен — иначе `401`), затем `requirePermission(..., "report.view")`
(хватает ли прав — иначе `403`).

Эндпоинты (база `/api`, как в спецификации):

| Метод | Путь | Право | Что отдаёт |
|-------|------|-------|------------|
| GET | `/api/reports/sales?from=&to=` | `report.view` | продажи за период (кол-во, выручка) |
| GET | `/api/reports/profit?from=&to=` | `report.view` | прибыль за период |
| GET | `/api/reports/top-products?from=&to=&limit=` | `report.view` | топ товаров |
| GET | `/api/reports/rating` | `report.view` | средняя оценка |
| GET | `/api/reports/demand-warnings?from=&to=&threshold=` | `report.view` | популярное и кончается |

```cpp
// src/api/report_controller.cpp
#include "report_controller.h"
#include "auth_middleware.h"     // requireAuth, requirePermission (шаги 10, 11)
#include "../../third_party/json.hpp"
using nlohmann::json;

namespace fs {

// Маленький помощник: достать query-параметр или вернуть значение по умолчанию.
static std::string q(const httplib::Request& req, const std::string& key,
                     const std::string& def) {
    return req.has_param(key) ? req.get_param_value(key) : def;
}

void ReportController::registerRoutes(httplib::Server& srv) {

    // GET /api/reports/sales?from=2026-06-01&to=2026-07-01
    srv.Get("/api/reports/sales", [this](const httplib::Request& req, httplib::Response& res) {
        int64_t userId;
        if (!requireAuth(req, res, authService_, userId)) return;             // -> 401
        if (!requirePermission(res, authzService_, userRepo_, userId, "report.view"))
            return;                                                           // -> 403

        // По умолчанию — «весь период»: с начала времён по далёкое будущее.
        std::string from = q(req, "from", "0000-01-01");
        std::string to   = q(req, "to",   "9999-12-31");

        SalesReport r = reportService_.sales(from, to);

        json out = {
            {"from", r.from}, {"to", r.to},
            {"orders_count", r.orders_count},
            {"revenue", r.revenue}
        };
        res.status = 200;
        res.set_content(out.dump(), "application/json");   // объект -> JSON (только в API!)
    });

    // GET /api/reports/profit?from=&to=
    srv.Get("/api/reports/profit", [this](const httplib::Request& req, httplib::Response& res) {
        int64_t userId;
        if (!requireAuth(req, res, authService_, userId)) return;
        if (!requirePermission(res, authzService_, userRepo_, userId, "report.view")) return;

        ProfitReport r = reportService_.profit(q(req, "from", "0000-01-01"),
                                               q(req, "to",   "9999-12-31"));
        json out = {{"from", r.from}, {"to", r.to},
                    {"revenue", r.revenue}, {"cost", r.cost}, {"profit", r.profit}};
        res.status = 200;
        res.set_content(out.dump(), "application/json");
    });

    // GET /api/reports/top-products?from=&to=&limit=10
    srv.Get("/api/reports/top-products", [this](const httplib::Request& req, httplib::Response& res) {
        int64_t userId;
        if (!requireAuth(req, res, authService_, userId)) return;
        if (!requirePermission(res, authzService_, userRepo_, userId, "report.view")) return;

        int limit = std::stoi(q(req, "limit", "10"));     // сколько строк вернуть
        auto rows = reportService_.topProducts(q(req, "from", "0000-01-01"),
                                               q(req, "to",   "9999-12-31"), limit);
        json arr = json::array();
        for (const TopProductRow& row : rows) {
            arr.push_back({{"product_id", row.product_id},
                           {"product_name", row.product_name},
                           {"sold_qty", row.sold_qty},
                           {"revenue", row.revenue}});
        }
        res.status = 200;
        res.set_content(arr.dump(), "application/json");
    });

    // GET /api/reports/rating
    srv.Get("/api/reports/rating", [this](const httplib::Request& req, httplib::Response& res) {
        int64_t userId;
        if (!requireAuth(req, res, authService_, userId)) return;
        if (!requirePermission(res, authzService_, userRepo_, userId, "report.view")) return;

        RatingReport r = reportService_.rating();
        json out = {{"avg_rating", r.avg_rating}, {"reviews_count", r.reviews_count}};
        res.status = 200;
        res.set_content(out.dump(), "application/json");
    });
}

} // namespace fs
```
