## 1. Самый каркас программы

Любой пример ниже подразумевает такой «скелет». Мы будем показывать только содержимое
`main`, но помните, что вокруг него есть вот это:

```cpp
#include <iostream>   // подключаем ввод-вывод (std::cout)

int main() {
    // ... сюда вставляем код примера ...
    return 0;         // 0 = успешное завершение
}
```

- `#include <...>` — «подключи готовый кусок стандартной библиотеки».
- `int main()` — точка входа, с неё начинается программа.
- `std::cout << ...` — печать в терминал; `std::endl` — перевод строки.
- `std::` — «фамилия» стандартной библиотеки (пространство имён `std`).

## 2. Типы данных, которые мы используем

В C++ у каждой переменной есть **тип** — какого рода данные она хранит. Мы намеренно
ограничимся небольшим набором, зафиксированным для проекта (см. соглашения в README).

| Тип | Что хранит | Где в проекте |
|-----|-----------|---------------|
| `int64_t` | большое целое (64 бита) | **все id**: id заказа, id товара, id клиента |
| `double` | дробное число | деньги: цена букета, сумма заказа (учебно) |
| `bool` | да/нет (`true`/`false`) | флаги: заблокирован ли пользователь (`is_blocked`) |
| `std::string` | строка текста | имена, логины, адреса, названия цветов |

```cpp
#include <iostream>
#include <cstdint>   // даёт типы int64_t — целые точного размера
#include <string>    // даёт std::string

int main() {
    int64_t order_id = 1001;          // id заказа: всегда int64_t (а не int!)
    double  total_price = 2499.90;    // сумма заказа в рублях (дробное)
    bool    is_blocked = false;       // пользователь не заблокирован
    std::string flower_name = "Роза"; // название цветка — текст

    std::cout << "Заказ #" << order_id
              << " на сумму " << total_price << " руб." << std::endl;
    return 0;
}
```

> **Почему именно `int64_t` для id?** Обычный `int` на некоторых системах вмещает только
> ~2 миллиарда. Заказов и записей со временем может стать очень много, поэтому для id мы
> берём 64-битное целое — это решение зафиксировано на весь проект.

> **Почему `double` для денег — учебно?** Дробные числа `double` хранятся приближённо,
> поэтому в настоящих финансовых системах деньги считают в **целых копейках** (`int64_t`).
> Мы используем `double` ради простоты обучения и **честно об этом помним**.

## 3. Переменные и `const`

Переменная — это «коробка с именем», где лежит значение. Иногда мы хотим запретить менять
значение после присвоения — для этого есть слово **`const`** («константа», неизменяемое).

```cpp
#include <iostream>
#include <cstdint>

int main() {
    int64_t stock = 50;          // обычная переменная: значение можно менять
    stock = stock - 1;           // продали один цветок — остаток уменьшился

    const double VAT_RATE = 0.20; // НДС 20% — это правило не меняется в ходе работы
    // VAT_RATE = 0.25;          // ← ошибка компиляции! const менять нельзя

    std::cout << "Остаток: " << stock << ", НДС: " << VAT_RATE << std::endl;
    return 0;
}
```

> Привычка помечать `const` всё, что не должно меняться, — признак аккуратного кода.
> Компилятор тогда **сам поймает** случайную попытку что-то испортить.

## 4. Функции

**Функция** — это кусок кода с именем, которому можно дать данные (**аргументы**) и который
может вернуть результат. Это способ не повторять одно и то же.

```cpp
#include <iostream>

// Функция: имя calcTotal, принимает цену и количество, возвращает сумму (double).
// Слева от имени — ТИП возвращаемого значения. В скобках — аргументы с типами.
double calcTotal(double price_each, int quantity) {
    return price_each * quantity;   // вернуть произведение
}

int main() {
    double sum = calcTotal(299.0, 3);  // три цветка по 299 руб.
    std::cout << "Итого: " << sum << " руб." << std::endl; // 897
    return 0;
}
```

> Имена методов в проекте мы пишем в стиле **camelCase**: `calcTotal`, `findById`,
> `createOrder` (см. соглашения README).

## 5. Ссылки и `const&` — зачем передавать «по ссылке»

Это одно из важнейших понятий C++. Когда вы передаёте аргумент в функцию **по значению**,
делается его **копия**. Копировать одно число — дёшево. А вот копировать целый заказ со
списком цветов — дорого и расточительно.

**Ссылка** (`&`) — это не копия, а «второе имя» того же объекта. Передав по ссылке, функция
работает с **оригиналом**, без копирования.

```cpp
#include <iostream>
#include <string>

// Передаём по значению: name — это КОПИЯ строки. Длинную строку копировать накладно.
void printByValue(std::string name) {
    std::cout << "Клиент: " << name << std::endl;
}

// Передаём по ссылке: &name — это сам оригинал, копии нет.
// const означает "я обещаю не менять оригинал". Это идеальный способ ЧИТАТЬ большой объект.
void printByConstRef(const std::string& name) {
    std::cout << "Клиент: " << name << std::endl;
    // name = "другое"; // ← ошибка: const запрещает менять
}

// Ссылка БЕЗ const — функция МОЖЕТ изменить оригинал. Используем, когда это нужно нарочно.
void addBonus(int& points) {
    points += 100;   // меняем переменную вызывающего кода
}

int main() {
    std::string client = "Анна";
    printByConstRef(client);     // дёшево: без копии, читаем

    int points = 0;
    addBonus(points);            // points станет 100 (изменили оригинал)
    std::cout << "Бонусы: " << points << std::endl;
    return 0;
}
```

**Правило, которое мы применяем во всём проекте:** большие объекты для **чтения** передаём
как **`const&`** (константная ссылка) — это и быстро (нет копии), и безопасно (нельзя
случайно испортить). Вы увидите `const Order&`, `const std::string&` повсюду.

## 6. `struct` против `class` и поля

И `struct`, и `class` позволяют объединить несколько переменных в **одну составную
сущность**. Технически разница ровно одна: в `struct` поля по умолчанию **публичные**
(доступны снаружи), в `class` — **приватные** (спрятаны).

В нашем проекте действует чёткое соглашение (из README и шага `03`):

- **Доменные сущности** (данные: `User`, `Order`, `Product`...) — это **`struct`** с
  публичными полями. Они просто **хранят данные**.
- **Сервисы, репозитории, контроллеры** (поведение) — это **`class`** с методами.

```cpp
#include <iostream>
#include <cstdint>
#include <string>

// Доменная сущность "Товар" — это struct. Поля публичные, имена в snake_case,
// совпадают с колонками таблицы в БД (см. шаг 04). Это просто "пакет данных".
struct Product {
    int64_t id;            // первичный ключ
    std::string name;      // название, например "Букет 'Весна'"
    double base_price;     // базовая цена
};

int main() {
    Product p;                 // создаём товар
    p.id = 7;                  // доступ к полю через точку
    p.name = "Букет 'Весна'";
    p.base_price = 1990.0;

    std::cout << p.name << " — " << p.base_price << " руб." << std::endl;
    return 0;
}
```

> Запомните соглашение: **поля сущностей — `snake_case`** (`base_price`, `full_name`,
> `is_blocked`), потому что они один-в-один повторяют имена колонок базы данных. А имена
> **типов** — `PascalCase` (`Product`, `Order`, `User`).

## 7. Пространства имён (`namespace fs`)

**Пространство имён** — это «фамилия» для имён, чтобы они не сталкивались. У стандартной
библиотеки фамилия `std`. **Весь наш код** живёт в пространстве имён **`fs`** (от
*flowershop*).

```cpp
#include <iostream>
#include <cstdint>
#include <string>

// Всё своё кладём в namespace fs — чтобы наши имена не конфликтовали с чужими.
namespace fs {
    struct Category {
        int64_t id;
        std::string name;
    };
}

int main() {
    fs::Category c;          // обращаемся через фамилию: fs::Category
    c.id = 1;
    c.name = "Розы";
    std::cout << "Категория: " << c.name << std::endl;
    return 0;
}
```

## 8. Заголовок `.h` против `.cpp` и `#pragma once`

В шаге `01` мы уже познакомились: `.h` — это **анонс** (что есть), `.cpp` — **реализация**
(как это сделано). Теперь увидим на коде. Это **главный приём** организации проекта.

**Файл `product.h`** (анонс — кладётся в `src/domain/`):

```cpp
// product.h
#pragma once          // "включи этот файл при сборке только ОДИН раз" — защита от дублей
#include <cstdint>
#include <string>

namespace fs {
    // Объявляем структуру: какие у неё поля. Реализации методов тут может не быть.
    struct Product {
        int64_t id;
        std::string name;
        double base_price;
    };
}
```

**Файл `main.cpp`** (использует анонс):

```cpp
// main.cpp
#include <iostream>
#include "product.h"   // КАВЫЧКИ "..." = ищи в моём проекте (а не <...> в стандартных)

int main() {
    fs::Product p{42, "Тюльпан", 99.0}; // создаём и сразу заполняем поля по порядку
    std::cout << p.name << ": " << p.base_price << std::endl;
    return 0;
}
```

> **Что такое `#pragma once`?** Один и тот же `.h` может быть подключён много раз (напрямую
> и через другие заголовки). `#pragma once` говорит компилятору «вставь меня только один
> раз» — иначе будут ошибки повторного определения. Все наши заголовки начинаются с него
> (соглашение README).

> **Кавычки `"..."` или скобки `<...>`?** `<httplib.h>`/`<iostream>` — для стандартных и
> сторонних библиотек (искать в системных путях). `"product.h"` — для **своих** файлов
> проекта.

## 9. Контейнеры: `vector`, `optional`, `map`

Стандартная библиотека даёт готовые «коллекции». Нам нужны три.

### 9.1. `std::vector` — список переменной длины

`vector<T>` — это «умный массив»: автоматически растёт, знает свой размер. В заказе может
быть произвольное число позиций — храним их в векторе.

```cpp
#include <iostream>
#include <vector>     // даёт std::vector
#include <string>

int main() {
    std::vector<std::string> bouquet;   // пустой список названий цветов
    bouquet.push_back("Роза");          // добавить в конец
    bouquet.push_back("Лилия");
    bouquet.push_back("Гипсофила");

    std::cout << "Цветов в букете: " << bouquet.size() << std::endl; // 3
    return 0;
}
```

### 9.2. `std::optional` — «значение есть или его нет»

`optional<T>` решает вечную проблему «а вдруг значения нет». Например, репозиторий ищет
пользователя по id — а вдруг такого нет? Вместо «выдуманного пустого User» вернём
**`optional<User>`**: либо значение, либо «пусто».

```cpp
#include <iostream>
#include <optional>   // даёт std::optional
#include <cstdint>

// Возвращает остаток на складе, ЕСЛИ товар существует; иначе — "пусто".
std::optional<int64_t> findStock(int64_t product_id) {
    if (product_id == 7) return 50;     // нашли: 50 штук
    return std::nullopt;                // не нашли: пусто
}

int main() {
    auto result = findStock(7);
    if (result.has_value()) {           // проверяем, есть ли значение
        std::cout << "Остаток: " << result.value() << std::endl;
    } else {
        std::cout << "Товар не найден" << std::endl;
    }
    return 0;
}
```

> Мы будем возвращать `std::optional<...>` из методов поиска в репозиториях (шаг `07`),
> чтобы честно выражать «может не найтись», не прибегая к опасным «магическим» значениям.

### 9.3. `std::map` — словарь «ключ → значение»

`map<K,V>` хранит пары: по **ключу** быстро достаём **значение**. Удобно, например, для
«id товара → сколько штук заказали».

```cpp
#include <iostream>
#include <map>        // даёт std::map
#include <cstdint>

int main() {
    std::map<int64_t, int> order_items;  // ключ: id товара, значение: количество
    order_items[7] = 3;                  // товара #7 заказали 3 шт.
    order_items[12] = 1;                 // товара #12 — 1 шт.

    std::cout << "Товара #7 заказано: " << order_items[7] << std::endl;
    return 0;
}
```

## 10. Цикл `for` по диапазону и `auto`

### 10.1. Диапазонный `for`

Чтобы пройтись по всем элементам контейнера, используем «`for` по диапазону» — он короткий
и без счётчиков-индексов.

```cpp
#include <iostream>
#include <vector>
#include <string>

int main() {
    std::vector<std::string> bouquet{"Роза", "Лилия", "Пион"};

    // "для каждого name из bouquet". const& — читаем без копирования (см. раздел 5).
    for (const std::string& name : bouquet) {
        std::cout << "- " << name << std::endl;
    }
    return 0;
}
```

### 10.2. `auto` — «выведи тип сам»

Иногда тип длинный и очевидный из контекста. Слово **`auto`** просит компилятор
**вывести тип автоматически**. Это сокращение, не «динамическая типизация» — тип всё равно
один и фиксируется на этапе компиляции.

```cpp
#include <iostream>
#include <vector>
#include <string>

int main() {
    std::vector<std::string> bouquet{"Роза", "Лилия"};

    // Вместо длинного "const std::string&" пишем auto — компилятор сам поймёт тип.
    for (const auto& name : bouquet) {
        std::cout << name << std::endl;
    }
    return 0;
}
```

> Совет новичку: используйте `auto`, когда тип и так очевиден из правой части (как в цикле
> по вектору строк). Если тип неочевиден — лучше написать его явно ради читаемости.

## 11. Немного ООП: методы и конструкторы

Пока сущности-данные у нас `struct` с полями. Но **классы поведения** (сервисы,
репозитории) имеют **методы** — функции внутри класса — и **конструктор** — особый метод,
который вызывается при создании объекта, чтобы его «настроить».

```cpp
#include <iostream>
#include <cstdint>

namespace fs {
    // Класс с поведением: считает цену заказа. Имя в PascalCase, суффикс Service.
    class PricingService {
    public:
        // Конструктор: вызывается при создании объекта. Принимает ставку наценки
        // и сохраняет её в поле markup_. Двоеточие — это "список инициализации".
        explicit PricingService(double markup) : markup_(markup) {}

        // Метод (функция внутри класса). camelCase. const после скобок означает
        // "этот метод не меняет поля объекта" — только читает.
        double finalPrice(double base_price) const {
            return base_price * (1.0 + markup_);
        }

    private:
        double markup_;   // приватное поле: наценка, например 0.15 = +15%
                          // (поля классов-сервисов помечаем _ на конце — это наш стиль)
    };
}

int main() {
    fs::PricingService pricing(0.15);            // создаём объект, конструктор сохранит 0.15
    std::cout << pricing.finalPrice(1000.0)      // вызов метода через точку → 1150
              << " руб." << std::endl;
    return 0;
}
```

> `public:` — что доступно снаружи (методы-«кнопки»). `private:` — что спрятано внутри
> (поля, детали). Это **инкапсуляция**: снаружи пользуются методами, не зная внутренностей.

## 12. `enum class` — для статусов

Статус заказа — это не произвольный текст, а **один из заранее известных вариантов**:
«новый», «оплачен», «собирается», «доставляется», «выполнен», «отменён». Для таких
«перечислений с фиксированным набором значений» в C++ есть **`enum class`**.

```cpp
#include <iostream>

namespace fs {
    // Перечисление статусов заказа. Имя типа PascalCase: OrderStatus.
    // "class" в enum class делает значения "строгими": их нельзя случайно
    // перепутать с числами или со значениями другого enum.
    enum class OrderStatus {
        New,          // новый
        Paid,         // оплачен
        Assembling,   // собирается флористом
        Delivering,   // в доставке
        Done,         // выполнен
        Cancelled     // отменён
    };
}

int main() {
    fs::OrderStatus status = fs::OrderStatus::New;  // обращение через имя enum::значение

    if (status == fs::OrderStatus::New) {
        std::cout << "Это новый заказ" << std::endl;
    }
    return 0;
}
```

> Почему не хранить статус строкой `"new"`? Потому что строку легко написать с опечаткой
> (`"nеw"`), и компилятор не заметит. А `enum class` проверяется компилятором: написали
> несуществующий статус — сразу ошибка сборки. В БД статус будет лежать как `status_id`
> (число), и мы будем превращать его в `OrderStatus` (это сделаем в шаге `06`).

## 13. Умные указатели (`unique_ptr`) — пока вскользь

Иногда объект надо создавать «в куче» (динамически) и кто-то должен его потом **удалить**,
чтобы не было утечки памяти. Раньше за этим следил программист вручную (и часто забывал).
Сейчас за нас это делает **умный указатель** — он автоматически удалит объект, когда тот
больше не нужен.

```cpp
#include <iostream>
#include <memory>     // даёт std::unique_ptr
#include <string>

namespace fs {
    struct Logger {
        void info(const std::string& msg) { std::cout << "[INFO] " << msg << "\n"; }
    };
}

int main() {
    // unique_ptr — "единоличный владелец" объекта Logger.
    // make_unique создаёт объект. Когда logger выйдет из области видимости —
    // объект Logger удалится САМ. Память чистить руками не надо.
    std::unique_ptr<fs::Logger> logger = std::make_unique<fs::Logger>();
    logger->info("Сервер запущен");   // у указателя обращение к методу через ->
    return 0;
}
```

> Пока запомните одно: `unique_ptr` — это «владелец, который сам приберёт за собой».
> Подробно (зачем именно так, чем отличается от `shared_ptr`, что такое «владение») мы
> разберём позже, когда будем собирать инфраструктуру в шагах `07`–`10`. Сейчас достаточно
> узнавать эту конструкцию в коде.

## 14. Обработка ошибок: `try`/`catch` (кратко)

Что делать, если «пошло не так» — товара нет, БД недоступна, данные кривые? В C++ для
**исключительных ситуаций** используют **исключения**: код «бросает» исключение
(`throw`), а вызывающая сторона его «ловит» (`try`/`catch`).

```cpp
#include <iostream>
#include <stdexcept>  // даёт готовые типы исключений, напр. std::runtime_error

// Бросает исключение, если количество отрицательное.
void checkQuantity(int quantity) {
    if (quantity < 0) {
        throw std::runtime_error("Количество не может быть отрицательным");
    }
    std::cout << "Количество в порядке: " << quantity << std::endl;
}

int main() {
    try {
        checkQuantity(-5);             // тут "взорвётся" исключение
    } catch (const std::exception& e) { // ловим по const& (см. раздел 5)
        std::cout << "Ошибка: " << e.what() << std::endl; // .what() — текст ошибки
    }
    return 0;
}
```

> В проекте сервисы будут **бросать** исключения при нарушении бизнес-правил («нет на
> складе», «дубль заказа»), а слой API будет их **ловить** и превращать в HTTP-коды
> (400, 409 и т.д.). Это мы соберём в шагах `08`–`09`.

## 15. Чего мы НЕ касаемся (специально)

Чтобы не перегружать новичка, мы **сознательно** оставляем на потом или не трогаем вовсе:

- **Шаблоны** (templates, `template<...>`) — мощный, но сложный механизм. Мы пользуемся
  готовыми шаблонными типами (`vector`, `optional`, `map`), но **свои** шаблоны писать не
  будем.
- **Move-семантика** (`std::move`, rvalue-ссылки) — оптимизация копирования. Упоминаем,
  что она существует и почему `unique_ptr` «нельзя копировать, можно только перемещать»,
  но углубляться не будем.
- **Множественное наследование, виртуальные базы, метапрограммирование** — не нужны для
  нашего проекта.

Этого набора из разделов 1–14 **достаточно**, чтобы прочитать и написать любой файл
оставшейся документации. Если впереди встретится что-то новое — мы объясним это на месте.
