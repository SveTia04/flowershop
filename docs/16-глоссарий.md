## A. Язык C++ и инструменты сборки

- **Компилятор** — программа, которая переводит ваш исходный код на C++ в машинный код
  (понятный процессору). Если в коде ошибка — именно компилятор «ругается». *(шаг `01`, `02`)*
- **Линковщик (компоновщик, linker)** — программа, которая **склеивает** скомпилированные
  кусочки (`.o`/`.obj`) и библиотеки в один исполняемый файл. Ошибка линковки ≠ ошибка
  компиляции: код собрался, но не нашлось чего-то для склейки (например, двух `main`). *(шаг `01`, `14`)*
- **Заголовочный файл (header, `.h`)** — файл с **объявлениями** (что существует: какие
  классы, какие функции). Подключается через `#include`. Это «оглавление/анонс». *(шаг `02`)*
- **Файл реализации (`.cpp`)** — файл с **самим кодом** функций, объявленных в заголовке.
  «Текст», к которому заголовок — «оглавление». *(шаг `02`)*
- **`#pragma once`** — строка в начале заголовка: «подключи меня не больше одного раза».
  Защита от двойного включения одного и того же файла. *(шаг `02`)*
- **`#include`** — директива «вставь сюда содержимое этого файла» (обычно заголовка). *(шаг `02`)*
- **CMake** — система сборки: по файлу `CMakeLists.txt` она готовит команды для компилятора
  и линковщика на любой ОС. Избавляет от ручного вызова `g++` с десятком флагов. *(шаг `05`, `14`, `15`)*
- **Цель сборки (target)** — «что мы собираем»: один исполняемый файл или библиотека. У нас
  две цели: `flowershop` (сервер) и `tests` (тесты). *(шаг `14`, `15`)*
- **Header-only библиотека** — библиотека из одних заголовков (`httplib.h`, `json.hpp`,
  `doctest.h`): подключил `#include` — и всё, отдельно линковать нечего. *(шаг `05`)*
- **Debug / Release** — режимы сборки. Debug — без оптимизаций, с отладочной информацией (для
  разработки). Release — оптимизированный, быстрый (для сдачи). *(шаг `15`)*
- **namespace (пространство имён)** — «папка для имён», чтобы наши `User`, `Order` не
  столкнулись с чужими. Весь наш код — в `namespace fs` (flowershop). *(шаг `02`)*

## B. Конструкции C++

- **`struct` / `class`** — способ описать **свой тип** (сущность) с полями (данными) и
  методами (действиями). У `struct` поля по умолчанию открытые — мы используем `struct` для
  доменных моделей (`User`, `Order`). *(шаг `02`, `06`)*
- **Поле** — переменная внутри `struct`/`class` (например, `Order::total_price`). *(шаг `06`)*
- **Метод** — функция внутри класса (например, `OrderService::createOrder`). *(шаг `09`)*
- **Конструктор** — особый метод, который выполняется при создании объекта; через него мы
  передаём зависимости (инъекция). *(шаг `09`)*
- **Ссылка (`&`, например `OrderRepository&`)** — «другое имя того же объекта», не копия.
  Передаём зависимости по ссылке, чтобы все работали с **одним** объектом. *(шаг `09`)*
- **`const` / `const&`** — обещание «я это не изменю». `const std::string&` — «дай посмотреть,
  копировать не буду и менять не буду» (быстро и безопасно). *(шаг `02`, `09`)*
- **Указатель** — переменная, хранящая **адрес** другого объекта (может быть «пустой» —
  `nullptr`). Более «низкоуровневый» брат ссылки. *(шаг `02`)*
- **`std::vector<T>`** — «растущий массив» элементов типа `T` (например, `vector<OrderItem>`
  — позиции заказа). Умеет `push_back` (добавить в конец). *(шаг `02`, `06`)*
- **`std::optional<T>`** — «значение, которого может и не быть»: либо есть `T`, либо пусто.
  `findById` возвращает `optional<Order>`: нашли — есть, не нашли — пусто. *(шаг `07`)*
- **`int64_t`** — целое число «пошире» (8 байт). Все `id` у нас этого типа. *(шаг `06`)*
- **`double`** — дробное число. Учебно храним в нём деньги (в проде — целые копейки). *(шаг `09`)*
- **Исключение (exception) / `throw`** — способ сообщить об ошибке «бросанием»: метод бросает
  `DuplicateOrderError`, а кто-то выше его «ловит». Контроллер превращает исключение в
  HTTP-код. *(шаг `08`, `09`)*
- **RAII** (Resource Acquisition Is Initialization) — приём C++: ресурс (открытая база,
  транзакция) захватывается в конструкторе объекта и **сам освобождается**, когда объект
  выходит из области видимости. Поэтому нам не надо вручную «закрывать» — `Transaction`
  откатится сам, если забыли `commit`. *(шаг `07`)*

## C. Архитектура проекта

- **Слой (layer)** — горизонтальный «этаж» системы. У нас четыре: API → Сервисы →
  Репозитории → БД. Правило: слой обращается только к слою **под** собой. *(шаг `03`)*
- **Модуль** — вертикальная «тема» (Заказы, Склад, Отчёты). На пересечении слоя и модуля —
  конкретный файл. *(шаг `03`)*
- **Домен (доменная модель)** — структуры данных, отражающие «вещи из жизни магазина»
  (`User`, `Product`, `Order`). Только данные, без поведения. *(шаг `06`)*
- **Репозиторий (repository)** — слой доступа к данным: умеет только читать/писать строки в
  БД через SQL. «Кладовщик». Здесь и **только** здесь живёт SQL. *(шаг `07`)*
- **Сервис (service)** — слой бизнес-логики: правила магазина («дубль нельзя», «хватает ли
  цветов», «посчитать цену»). «Повар». Ни SQL, ни HTTP. *(шаг `09`)*
- **Контроллер (controller)** — слой API: принимает HTTP-запрос, разбирает JSON, зовёт
  сервис, отдаёт JSON и HTTP-код. «Официант». *(шаг `08`)*
- **Инфраструктура (infra)** — служебное «сбоку»: подключение к БД, безопасность, логи,
  конфиг. *(шаг `03`, `10`, `15`)*
- **Инъекция зависимостей (Dependency Injection)** — приём: помощников (репозитории) **выдают
  сервису снаружи** через конструктор, а не создают внутри. Делает код тестируемым. *(шаг `09`, `14`)*
- **Транзакция (в смысле БД)** — группа изменений «всё-или-ничего»: либо все применились
  (`COMMIT`), либо ни одного (`ROLLBACK`). Чтобы не осталось «полузаказа». *(шаг `07`, `09`)*

## D. База данных и SQL

- **База данных (БД)** — организованное хранилище данных. У нас — SQLite. *(шаг `04`)*
- **SQLite** — встраиваемая СУБД: вся база — это **один файл** `flowershop.db`, без
  отдельного сервера. *(шаг `04`)*
- **PostgreSQL** — «взрослая» серверная реляционная СУБД; на неё можно перейти, поменяв только
  нижний слой (намёк ТЗ 3.6.1). *(шаг `15`)*
- **Реляционная СУБД** — база, где данные лежат в **таблицах**, связанных между собой. *(шаг `04`)*
- **Таблица** — данные одной сущности, как лист Excel: строки (объекты) и столбцы (свойства). *(шаг `04`)*
- **Строка (row) / столбец (column, поле)** — один объект / одно его свойство. *(шаг `04`)*
- **Первичный ключ (PRIMARY KEY, `id`)** — уникальный номер строки, по нему находят объект. *(шаг `04`)*
- **Внешний ключ (FOREIGN KEY)** — поле, ссылающееся на `id` в другой таблице (например,
  `orders.client_id` → `users.id`). Так таблицы связываются. *(шаг `04`)*
- **Целостность данных** — гарантии, что в базе нет «мусора»: нельзя заказ без клиента,
  рейтинг 7, два одинаковых логина. Обеспечивают ограничения `NOT NULL`, `FOREIGN KEY`,
  `UNIQUE`, `CHECK`, `DEFAULT`. *(шаг `04`)*
- **SQL** — язык запросов к реляционной БД («дай», «вставь», «обнови»). *(шаг `04`, `07`)*
- **Агрегация** — сворачивание многих строк в итог: `SUM` (сумма), `COUNT` (количество),
  `AVG` (среднее), `GROUP BY` (по группам). Основа отчётов. *(шаг `13`)*
- **`JOIN`** — соединение таблиц в запросе: «приклей к заказу данные его клиента». *(шаг `13`)*
- **CTE (`WITH ... AS`)** — временная «таблица на один запрос», для читаемости сложных
  запросов (мы так считаем среднюю закупочную цену в отчёте прибыли). *(шаг `13`)*
- **Транзакция** — см. раздел C (это понятие на стыке кода и БД). *(шаг `07`)*
- **SQL-инъекция** — атака: злоумышленник вписывает кусок SQL в поле ввода, чтобы база
  выполнила **его** команду. Защита — никогда не склеивать SQL из строк. *(шаг `07`, `10`)*
- **Prepared statement (подготовленный запрос)** — запрос с «дырками» (`?`), куда значения
  **подставляются отдельно** (`bind`), а не склеиваются. Главная защита от SQL-инъекций. *(шаг `07`, `13`)*
- **`:memory:`** — особое «имя файла» SQLite: база живёт в оперативной памяти и исчезает при
  закрытии. Используем для изолированных тестов. *(шаг `14`)*
- **Бэкап (резервная копия)** — копия базы на случай поломки; для SQLite — копия файла
  командой `.backup`. *(шаг `15`)*

## E. Сеть, HTTP и REST API

- **HTTP** — протокол «вопрос-ответ» между клиентом и сервером. Клиент шлёт запрос — сервер
  отвечает. *(шаг `08`)*
- **Сервер** — программа, которая ждёт запросы и отвечает (наш cpp-httplib). *(шаг `08`)*
- **Клиент (API)** — кто шлёт запросы: браузер, `curl`, Postman, мобильное приложение. *(шаг `08`)*
- **REST** — стиль построения API: ресурсы по адресам (`/api/orders/42`) + HTTP-методы как
  глаголы. *(шаг `08`)*
- **Эндпоинт (endpoint)** — конкретный адрес+метод API (например, `POST /api/orders`). *(шаг `08`)*
- **HTTP-метод** — глагол запроса: `GET` (прочитать), `POST` (создать), `PATCH` (изменить),
  `DELETE` (удалить). *(шаг `08`)*
- **Код состояния (HTTP status)** — число-результат: `200` ОК, `201` создано, `400` плохой
  запрос, `401` не авторизован, `403` нет прав, `404` не найдено, `409` конфликт (дубль),
  `500` ошибка сервера. *(шаг `08`)*
- **JSON** — текстовый формат обмена данными (`{"id": 1, "name": "роза"}`). На нём общаются
  клиент и наш API. *(шаг `08`)*
- **nlohmann/json** — C++-библиотека для работы с JSON (`nlohmann::json`). *(шаг `08`)*
- **cpp-httplib** — header-only C++-библиотека HTTP-сервера (`httplib.h`). *(шаг `08`)*
- **curl** — консольная утилита, шлёт HTTP-запрос одной командой; удобно для проверки API. *(шаг `08`, `14`)*
- **Postman** — программа с окошками для ручного составления и сохранения HTTP-запросов. *(шаг `14`)*
- **Обратный прокси (reverse proxy)** — сервер-«охранник» (nginx) перед нашим: принимает
  HTTPS снаружи, передаёт HTTP нашему серверу внутри. *(шаг `15`)*
- **TLS / HTTPS** — шифрование передачи данных по сети (требование ТЗ 3.4). *(шаг `15`)*

## F. Безопасность, аутентификация, доступ

- **Аутентификация** — установление «кто ты» (вход по логину/паролю → токен). *(шаг `10`)*
- **Авторизация** — установление «что тебе можно» (хватает ли прав на действие). *(шаг `11`)*
- **Токен** — «электронный пропуск», который сервер выдаёт после входа; клиент предъявляет
  его в заголовке `Authorization: Bearer <токен>`. *(шаг `08`, `10`)*
- **Хеш (hash)** — необратимый «отпечаток» пароля. В базе хранят его, а не сам пароль. У нас
  учебно SHA-256. *(шаг `10`)*
- **Соль (salt)** — случайная добавка к паролю перед хешированием, чтобы одинаковые пароли
  давали разные хеши и нельзя было сверить по готовым таблицам. *(шаг `10`)*
- **RBAC** (Role-Based Access Control) — управление доступом через роли: у человека роль, у
  роли — набор прав. *(шаг `11`)*
- **Роль** — должность в системе: `admin`, `florist`, `seller`, `courier`, `purchaser`,
  `client`, `owner`. *(шаг `11`)*
- **Право (permission)** — разрешение на действие, вида `order.create`, `report.view`. *(шаг `11`)*
- **Middleware** — «прослойка»-проверка перед обработкой запроса. У нас `requireAuth`
  (есть ли токен → иначе `401`) и `requirePermission` (хватает ли прав → иначе `403`). *(шаг `10`, `11`)*
- **Аудит / `audit_log`** — журнал «кто, что и когда сделал» (требование ТЗ для админа). *(шаг `12`)*

## G. Тестирование и качество

- **Тест** — маленькая программа, проверяющая, что код выдаёт ожидаемый результат. *(шаг `14`)*
- **Юнит-тест (unit test)** — тест одной маленькой детали (одного метода) в изоляции. *(шаг `14`)*
- **Регрессия** — поломка ранее работавшего после изменений; тесты её ловят автоматически. *(шаг `14`)*
- **doctest** — header-only C++-библиотека для тестов (`TEST_CASE`, `CHECK`, `REQUIRE`). *(шаг `14`)*
- **`CHECK` / `REQUIRE` / `CHECK_THROWS_AS`** — проверки doctest: мягкая / жёсткая (прерывает
  тест) / «должно бросить такое исключение». *(шаг `14`)*
- **ctest** — запускалка тестов из CMake (`ctest --output-on-failure`). *(шаг `14`)*

## H. Предметная область (цветочный магазин)

- **Заказ (`orders`) / позиция заказа (`order_items`)** — что заказал клиент / отдельная
  строка заказа (товар + количество + цена за штуку `price_each`). *(шаг `04`, `09`)*
- **Товар (`products`)** — цветок, букет или расходник (`kind` = `flower`/`bouquet`/`supply`),
  с ценой `base_price`. *(шаг `04`)*
- **Остаток (`stock`) / списание (`write_offs`)** — сколько товара на складе по партиям /
  запись о том, что товар ушёл (причина `использован`/`просрочка`/...). *(шаг `04`, `09`)*
- **Поступление (`supplies` / `supply_items`)** — партия от поставщика; в позиции есть
  закупочная цена `purchase_price` (нужна для отчёта прибыли). *(шаг `04`, `13`)*
- **Срок годности (`expires_at`)** — дата, после которой товар списывается автоматически
  (учёт сроков годности из ТЗ). *(шаг `04`, `09`)*
- **Статус заказа (`order_statuses`)** — этап жизни заказа: `new`, `paid`, `assembling`,
  `ready`, `delivering`, `done`, `cancelled`, `returned`. *(шаг `04`, `09`)*
- **Дубль заказа** — повторный одинаковый заказ того же клиента за короткое время; система
  его запрещает (`409 Conflict`) — требование ТЗ. *(шаг `09`, `14`)*
- **Отзыв (`reviews`) / рейтинг (`rating`)** — оценка заказа клиентом 1–5; основа отчёта
  «средняя оценка». *(шаг `04`, `13`)*
- **Уведомление (`notifications`)** — сообщение пользователю о событии (новый заказ, мало
  товара, смена статуса). *(шаг `04`, `12`)*
- **Отчёт** — агрегированный взгляд на данные: продажи, прибыль, топ товаров (для владельца). *(шаг `13`)*

## Как пользоваться этим глоссарием (для самопроверки)

1. **Накрой определение, оставь термин.** Прочитайте только слово — попробуйте объяснить его
   сами. Не получилось — открыли определение, перешли по ссылке на шаг.
2. **От термина — к месту в коде.** Для каждого термина из разделов C, D, H найдите его в
   соответствующем шаге и покажите пальцем строку кода/SQL, где он «живёт».
3. **Свяжите пары.** Аутентификация ↔ авторизация; сервис ↔ репозиторий; `CHECK` ↔ `REQUIRE`;
   хеш ↔ соль; `SUM` ↔ `GROUP BY`. Объясните разницу в каждой паре — это любимые вопросы на
   защите проекта.
