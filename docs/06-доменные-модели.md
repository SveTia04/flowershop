# Шаг 06. Доменные модели

> **Цель шага:** превратить таблицы из схемы БД (шаг `04`) в **C++-структуры** — по одной
> на каждую сущность. В конце у вас будет папка `src/domain/` с заголовочными файлами, в
> которых описано, **какие данные** живут в системе. Это слой «только данные, без
> поведения»: структуры ничего не умеют делать, они просто хранят значения. Поведение
> (сохранить, посчитать, проверить) появится позже — в репозиториях и сервисах.

---

## 1. Что такое «доменная модель» и зачем она

Слово «домен» (domain) здесь означает «предметная область» — мир цветочного магазина:
клиенты, заказы, цветы, поставки. **Доменная модель** — это набор C++-типов, которые
описывают объекты этого мира на языке программы.

Простыми словами: в базе данных строка таблицы `users` — это запись из ячеек. Но C++ не
умеет работать с «строкой таблицы» напрямую. Ему нужен **тип**, у которого есть понятные
поля: `login`, `full_name`, `role_id`. Вот этот тип мы и создаём.

> **Бытовая аналогия.** Таблица в БД — это анкета на бумаге, разлинованная на графы.
> Доменная модель — это та же анкета, но «оживлённая» в памяти программы: к каждой графе
> можно обратиться по имени (`user.login`), передать её в функцию, положить в список.

Зачем выделять модели в отдельный слой:

- **Общий язык.** И репозиторий (который читает из БД), и сервис (бизнес-правила), и
  контроллер (API) говорят об одном и том же `User`. Не нужно у каждого свой формат.
- **Безопасность типов.** Компилятор не даст случайно положить телефон в поле цены —
  типы не совпадут. Это ловит ошибки ещё до запуска.
- **Чистота слоёв.** Модель не знает ни про SQL, ни про HTTP, ни про JSON. Поэтому её
  можно использовать где угодно, и она ни от чего не зависит.

## 2. Главное решение: `struct`, а не `class`

В C++ есть два почти одинаковых ключевых слова для создания своих типов: `struct` и
`class`. Разница ровно одна:

- в `struct` поля по умолчанию **публичные** (доступны снаружи: `user.login`);
- в `class` поля по умолчанию **приватные** (снаружи не видны, нужны методы-обёртки).

> **Аналогия.** `struct` — это открытая коробка с подписанными отделениями: бери что
> хочешь. `class` — это сейф: внутрь только через специальную дверцу (методы).

Наши доменные сущности — это **просто данные**. У них нет секретов и нет поведения,
которое надо прятать. Поэтому мы берём `struct`: меньше писанины, всё на виду, и это
честно отражает идею «модель — это набор полей».

```cpp
// Идея в двух строчках:
struct User {           // struct => поля публичные
    int64_t id;         // обращаемся напрямую: user.id
    std::string login;  // и так далее
};
```

Если бы мы взяли `class`, пришлось бы для каждого поля писать «геттер» и «сеттер» —
десятки лишних строк без всякой пользы. Для чистых данных это не нужно.

> **Запомните правило из шага `03`:** «модель — это данные, сервис — это поведение».
> Поэтому в наших `struct` **не будет методов** вроде `save()` или `calculateTotal()`.
> Сохранять `User` будет `UserRepository` (шаг `07`), а считать сумму заказа —
> `OrderService` (шаг `09`).

## 3. Почему поля называются в `snake_case` и совпадают с БД

Посмотрите на колонку в `schema.sql`: `password_hash`, `full_name`, `role_id`. Это стиль
**snake_case** — слова через подчёркивание. В C++ чаще пишут `camelCase` (`fullName`),
но мы сознательно делаем поля моделей **точно как колонки в БД**.

Причина практическая. На шаге `07` репозиторий будет копировать данные из строки
результата SQL в поля структуры. Когда имена совпадают **буква в букву**, код читается
без головной боли:

```cpp
// Колонка БД называется full_name — и поле структуры называется full_name.
// Глаз не спотыкается, ошибиться трудно:
user.full_name = читаемТекст(строка, "full_name");
```

Если бы поле звалось `fullName`, а колонка `full_name`, пришлось бы каждый раз держать в
голове «перевод» и легко перепутать. Один стиль на стыке БД ↔ модель = меньше ошибок.
Поэтому: **имена полей моделей = имена колонок из шага `04`, не меняем их.**

> Имена *типов* при этом пишутся в `PascalCase` (`User`, `OrderItem`) — это общепринятый
> для C++ стиль имён типов. То есть: тип `User`, а поля внутри — `snake_case`.

## 4. Какие типы C++ берём для колонок БД

В SQLite колонки бывают `INTEGER`, `REAL`, `TEXT`. Переведём их в типы C++ и сразу
объясним каждый выбор. Если какая-то конструкция встречается впервые — разбираем её.

### 4.1. Целые числа и идентификаторы — `int64_t`

Все `id` (и внешние ключи `role_id`, `order_id`, …) — это `INTEGER PRIMARY KEY` в
SQLite. Внутри SQLite такой ключ — это **64-битное** целое. Чтобы гарантированно
вместить любое значение, мы берём тип `int64_t` (целое ровно на 64 бита).

```cpp
#include <cstdint>   // отсюда берётся int64_t

int64_t id;          // -9 223 372 036 854 775 808 ... +9 223 372 036 854 775 807
```

Почему не обычный `int`? Размер `int` зависит от платформы (часто всего 32 бита — до
~2 миллиардов). Для счётчика заказов это, может, и хватит, но `int64_t` всегда один и
тот же на любой машине. Предсказуемость важнее экономии. Можно писать и `long long` —
это то же самое, но `int64_t` яснее говорит «ровно 64 бита».

Обычные счётные поля — `quantity` (количество), `rating` (оценка), `shelf_life_days` —
это небольшие числа, для них хватает обычного `int`.

### 4.2. Текст — `std::string`

Колонки `TEXT` (логин, имя, сообщение, дата как текст) хранятся в C++ как `std::string`
— это «умная строка», которая сама управляет памятью под текст.

```cpp
#include <string>

std::string login;   // строка любой длины, память под неё выделяется автоматически
```

> Заметьте: даты (`created_at`, `expires_at`, `delivery_at`) в нашей схеме хранятся как
> `TEXT` в формате `'2026-06-19 14:30:00'`. Поэтому в моделях это **тоже `std::string`**,
> а не какой-то специальный тип даты. Так проще для новичка и совпадает с тем, что
> реально лежит в БД. Разбор даты понадобится только при подсчётах (например, просрочка),
> и этим займётся сервис.

### 4.3. Деньги — `double` (с честной оговоркой про копейки)

Колонки `base_price`, `total_price`, `price_each`, `amount`, `purchase_price` — это
`REAL` (дробное число). В C++ берём `double` — число с плавающей точкой двойной
точности.

```cpp
double base_price;   // 1990.0, 149.99 и т.п.
```

> **ВАЖНАЯ оговорка — обязательно её усвойте.** `double` хранит дроби **приближённо**.
> Классический пример: `0.1 + 0.2` в `double` даёт `0.30000000000000004`, а не ровно
> `0.3`. Для учебного проекта это терпимо, но в **настоящих** финансовых системах деньги
> так не хранят — иначе копейки «уплывают». Правильный приём в проде: хранить деньги в
> **целых копейках** (`int64_t`), то есть `19.90 ₽` = `1990` копеек, и делить на 100 лишь
> при показе пользователю. Мы оставляем `double` ради простоты, но вы должны знать, что
> это сознательное учебное упрощение.

### 4.4. Логические значения — `bool`

В SQLite нет отдельного типа «да/нет», поэтому флаги (`is_blocked`, `is_active`,
`is_read`) хранятся как `INTEGER` со значением `0` или `1`. В C++ для этого есть
естественный тип `bool` (true/false).

```cpp
bool is_blocked;   // в БД 0/1, в C++ false/true
```

Перевод `0/1` ↔ `false/true` будет делать репозиторий при чтении строки (шаг `07`).
В модели мы держим уже «человеческий» `bool`.

### 4.5. Поля, которые могут быть пустыми (NULL) — `std::optional`

В БД некоторые поля могут быть **NULL** — «значения нет». Примеры из схемы:

- `orders.client_id` — NULL, если это продажа без регистрации клиента;
- `orders.seller_id`, `orders.delivery_at` — могут отсутствовать;
- `products.flower_type_id` — NULL для букета (у букета нет одного «вида цветка»);
- `deliveries.delivered_at`, `payments.paid_at` — пусто, пока не доставлено/не оплачено.

Как выразить «значение либо есть, либо его нет» в C++? Для этого в C++17 есть
`std::optional<T>` — «коробочка, в которой либо лежит значение типа `T`, либо ничего».

```cpp
#include <optional>

std::optional<int64_t> client_id;   // либо номер клиента, либо «пусто» (NULL)

// Как с ним работать:
if (client_id.has_value()) {        // есть ли значение?
    int64_t c = client_id.value();  // достаём его (или короче: *client_id)
} else {
    // здесь client_id пуст — это была продажа без регистрации
}
```

> **Зачем это лучше «магических» значений.** Можно было бы класть `-1` в `client_id` и
> считать, что «-1 значит нет клиента». Но это ловушка: легко забыть проверку и
> обработать `-1` как настоящий id. `std::optional` заставляет **явно** спросить «а есть
> ли значение?» — и компилятор/код напоминают об этом. Это честное отражение NULL из БД.

Правило простое: **колонка может быть NULL → поле модели оборачиваем в `std::optional`.**
Колонка `NOT NULL` → обычный тип без обёртки.

### 4.6. Списки вложенных объектов — `std::vector`

У заказа много позиций (`order_items`), у букета — много составляющих (`bouquet_items`).
«Много однотипных объектов» в C++ хранят в `std::vector<T>` — это автоматически
растущий массив.

```cpp
#include <vector>

std::vector<OrderItem> items;   // ноль, одна или много позиций заказа
items.push_back(один_item);     // добавить ещё одну
```

Подробнее, как `Order` собирает в себя свои `OrderItem`, — в разделе 7.

## 5. Фиксированные наборы значений: `enum class` или строка?

Некоторые колонки в схеме ограничены **списком допустимых значений** через `CHECK`:

- `supply_items.quality` и `stock.quality` — `'отличное' | 'хорошее' | 'среднее'`;
- `payments.method` — `'наличные' | 'карта' | 'онлайн'`;
- `payments.status` — `'ожидает' | 'оплачен' | 'возврат'`;
- `deliveries.status` — `'назначена' | 'в пути' | 'доставлена' | 'не доставлена'`;
- `products.kind` — `'flower' | 'bouquet' | 'supply'`.

Для таких «фиксированных наборов» в C++ есть идеальный инструмент — `enum class`
(перечисление). Это тип, у которого может быть только одно из заранее названных значений:

```cpp
enum class DeliveryStatus {   // тип «статус доставки»
    Assigned,                 // назначена
    InTransit,                // в пути
    Delivered,                // доставлена
    Failed                    // не доставлена
};

DeliveryStatus s = DeliveryStatus::InTransit;  // только одно из четырёх, иначе не скомпилируется
```

**Чем `enum class` лучше обычной строки:**

- **Опечатка ловится компилятором.** Напишете `"в путии"` строкой — программа примет и
  сломается позже. А `DeliveryStatus::InTransi` просто не скомпилируется.
- **Видны все варианты.** Открыл `enum` — сразу видишь полный список состояний.
- **Быстрее и компактнее** строки (это просто число под капотом).

### Наш осознанный выбор

Мы поступаем так, чтобы было и удобно, и согласованно со схемой:

1. Для **двух главных справочников-состояний** — роли (`roles`) и статусы заказа
   (`order_statuses`) — в БД есть **отдельные таблицы** со своими `id`. Значит, в моделях
   `User` и `Order` мы храним **`role_id` / `status_id` как `int64_t`** (ссылку на строку
   справочника), а сами справочники описываем структурами `Role` и `OrderStatus`. Так
   модель точно повторяет схему БД.

2. Для коротких наборов, заданных прямо через `CHECK` в колонке (качество, способ и
   статус оплаты, статус доставки, вид товара `kind`), мы **заведём `enum class`** ради
   безопасности типов, а рядом дадим функции перевода `enum ↔ строка` (текст для БД).
   Где для новичка нагляднее оставить просто строку — оставим строку и так и пометим.

> Коротко: **есть отдельная таблица-справочник → храним её `id`** (число-ссылка). **Набор
> зашит в `CHECK` колонки → `enum class`** с переводом в строку для записи в БД.

Перевод `enum ↔ строка` понадобится репозиторию (шаг `07`), потому что в БД эти значения
лежат текстом. Покажем такой перевод на примере качества в файле `stock.h`.

## 6. Заголовочные файлы доменных моделей

Каждая модель — это отдельный заголовочный файл (`.h`) в папке `src/domain/`. В начале
каждого файла стоит `#pragma once` — это указание компилятору «подключи этот файл только
один раз», чтобы при многократном `#include` не появлялись дубли определений.

Все типы лежат в `namespace fs { ... }` — это «пространство имён» проекта (flowershop).
Оно как фамилия: чтобы наш `User` не столкнулся с каким-нибудь чужим `User` из библиотеки.
Снаружи к типу обращаются как `fs::User`.

Дальше — все файлы по порядку.

### 6.1. `src/domain/role.h` — роль пользователя

```cpp
#pragma once
#include <cstdint>   // int64_t
#include <string>    // std::string

namespace fs {

// Справочник ролей: соответствует таблице roles (id, code, name).
// Примеры: code='admin' name='Администратор', code='florist' name='Флорист'.
struct Role {
    int64_t     id;     // PRIMARY KEY из таблицы roles
    std::string code;   // машинный код роли: 'admin', 'florist', 'seller', ...
    std::string name;   // человекочитаемое имя: 'Администратор'
};

} // namespace fs
```

### 6.2. `src/domain/order_status.h` — статус заказа

```cpp
#pragma once
#include <cstdint>
#include <string>

namespace fs {

// Справочник статусов заказа: таблица order_statuses (id, code, name).
// Примеры: 'new'/'Новый', 'paid'/'Оплачен', 'done'/'Выполнен', 'cancelled'/'Отменён'.
struct OrderStatus {
    int64_t     id;     // PRIMARY KEY из order_statuses
    std::string code;   // 'new', 'paid', 'assembling', 'ready', 'delivering', ...
    std::string name;   // 'Новый', 'Оплачен', ...
};

} // namespace fs
```

### 6.3. `src/domain/user.h` — пользователь (сотрудник или клиент)

```cpp
#pragma once
#include <cstdint>
#include <string>

namespace fs {

// Соответствует таблице users.
// Один тип и для сотрудников, и для зарегистрированных клиентов — их различает role_id.
struct User {
    int64_t     id;             // users.id
    std::string login;          // users.login — уникальный логин (UNIQUE в БД)
    std::string password_hash;  // users.password_hash — хеш пароля (НЕ сам пароль!)
    std::string password_salt;  // users.password_salt — «соль» к паролю (см. шаг 10)
    std::string full_name;      // users.full_name — ФИО
    int64_t     role_id;        // users.role_id — ссылка на roles.id (NOT NULL)
    bool        is_blocked;     // users.is_blocked — в БД 0/1, у нас false/true
    std::string created_at;     // users.created_at — дата создания как текст
};

} // namespace fs
```

> Обратите внимание: `password_hash` и `password_salt` — это **не** пароль в открытом
> виде. Пароль мы никогда не храним; храним его необратимый хеш. Подробно — на шаге `10`.

### 6.4. `src/domain/category.h` — категория товара

### 6.6. `src/domain/product.h` — товар (цветок, букет или расходник)

В этом файле впервые появляется `enum class` для колонки `kind`. Колонка
`flower_type_id` может быть NULL (у букета нет одного вида цветка) — поэтому
`std::optional`.

```cpp
#pragma once
#include <cstdint>
#include <string>
#include <optional>

namespace fs {

// Вид товара. В БД это колонка products.kind с CHECK ('flower','bouquet','supply').
// Берём enum class, чтобы компилятор не дал указать несуществующий вид.
enum class ProductKind {
    Flower,   // 'flower'  — отдельный цветок
    Bouquet,  // 'bouquet' — готовый букет (состоит из flower_type через bouquet_items)
    Supply    // 'supply'  — расходник/упаковка
};

// Соответствует таблице products.
struct Product {
    int64_t                id;              // products.id
    std::string            name;            // products.name
    std::optional<int64_t> category_id;     // products.category_id (может быть NULL)
    std::optional<int64_t> flower_type_id;  // NULL для букета/расходника
    ProductKind            kind;            // products.kind как enum
    double                 base_price;      // products.base_price (деньги — double, см. п.4.3)
    bool                   is_active;       // products.is_active — 0/1 -> bool
};

} // namespace fs
```

### 6.7. `src/domain/bouquet_item.h` — состав букета

```cpp
#pragma once
#include <cstdint>

namespace fs {

// Таблица bouquet_items: какие цветы и сколько входят в конкретный букет.
struct BouquetItem {
    int64_t id;                  // bouquet_items.id
    int64_t bouquet_product_id;  // ссылка на products.id (сам букет)

    int     quantity;            // bouquet_items.quantity — сколько штук этого цветка
};

} // namespace fs
```

### 6.8. `src/domain/supplier.h` — поставщик

`phone` и `contact` в схеме без `NOT NULL` — значит могут быть NULL — оборачиваем в
`std::optional`.

```cpp
#pragma once
#include <cstdint>
```

### 6.10. `src/domain/stock.h` — остатки на складе (+ enum качества)

Этот файл — «дом» для `enum class Quality`, так как качество впервые относится к складу.
Его подключают `supply.h` и `write_off`-логика.

```cpp
#pragma once
#include <cstdint>
#include <string>
#include <optional>

namespace fs {

// Качество товара. В БД это колонки stock.quality / supply_items.quality
// с CHECK ('отличное','хорошее','среднее'). Берём enum class ради безопасности.
enum class Quality {
    Excellent,  // 'отличное'
    Good,       // 'хорошее'
    Average     // 'среднее'
};

// --- Перевод enum <-> строка БД (понадобится репозиторию на шаге 07) ---
// Из enum в текст, который реально лежит в колонке:
inline std::string qualityToDb(Quality q) {
    switch (q) {
        case Quality::Excellent: return "отличное";
        case Quality::Good:      return "хорошее";
        case Quality::Average:   return "среднее";
    }
    return "среднее"; // защитное значение по умолчанию
}

// Из текста БД обратно в enum:
inline Quality qualityFromDb(const std::string& s) {
    if (s == "отличное") return Quality::Excellent;
    if (s == "хорошее")  return Quality::Good;
    return Quality::Average; // 'среднее' и всё прочее
}

// Соответствует таблице stock — текущие остатки по товару (партии).
struct Stock {
    int64_t                    id;          // stock.id
    int64_t                    product_id;  // ссылка на products.id
    int                        quantity;    // stock.quantity — сколько штук на складе
    Quality                    quality;     // stock.quality как enum
    std::optional<std::string> expires_at;  // срок годности (текст) или NULL
};

} // namespace fs
```

> `inline` у функций перевода нужно потому, что они определены прямо в заголовке. Без
> `inline` при подключении файла в несколько `.cpp` компоновщик ругался бы на «функцию
> определили дважды». `inline` разрешает такое определение в заголовке.

## 7. Как `Order` содержит свои `OrderItem`

Это самый важный для понимания момент шага. В БД заказ и его позиции лежат в **двух
разных таблицах**: `orders` (шапка заказа) и `order_items` (строки). Связь — через поле
`order_items.order_id`.

В C++ нам удобно иметь **один цельный объект** `Order`, внутри которого уже лежат все его
позиции. Для этого в `Order` есть поле:

```cpp
std::vector<OrderItem> items;   // массив позиций, принадлежащих этому заказу
```

`std::vector` — это динамический массив: в нём может быть 0, 1 или 100 элементов, и он
сам растёт по мере добавления. Каждый элемент — это `OrderItem` (одна строка заказа).

> **Аналогия.** `Order` — это папка-скоросшиватель. `items` — листы внутри неё. Сама
> папка (шапка: кто клиент, статус, сумма) одна, а листов-позиций может быть сколько
> угодно.

Как это собирается на практике (это будет делать `OrderRepository` на шаге `07`):

```cpp
// 1) Прочитали шапку заказа из таблицы orders:
fs::Order order;
order.id          = 1001;
order.client_id   = 42;          // optional примет число как «значение есть»
order.status_id   = 1;
order.total_price = 0.0;         // посчитает сервис

// 2) Прочитали все строки order_items, где order_id = 1001, и сложили в вектор:
fs::OrderItem it1{ /*id*/ 1, /*order_id*/ 1001, /*product_id*/ 7, /*quantity*/ 3, /*price_each*/ 199.0 };
fs::OrderItem it2{ /*id*/ 2, /*order_id*/ 1001, /*product_id*/ 9, /*quantity*/ 1, /*price_each*/ 990.0 };

order.items.push_back(it1);      // добавили первую позицию
order.items.push_back(it2);      // добавили вторую

// 3) Теперь весь заказ — это один объект. Пройтись по позициям можно так:
for (const fs::OrderItem& it : order.items) {
    // const fs::OrderItem& — это «ссылка только для чтения»:
    //   & = работаем с тем же объектом, не делая лишнюю копию (быстрее);
    //   const = обещаем не менять его (защита от случайной правки).
    // здесь доступны it.product_id, it.quantity, it.price_each ...
}
```

Точно так же `Supply` содержит `std::vector<SupplyItem> items` — поступление и его
позиции собираются в один объект. Это типовой приём: **«родитель» хранит вектор
«детей»**.

> Заметьте ещё раз: `Order` **не считает** свою сумму и **не сохраняет** себя. Он только
> хранит позиции. Подсчёт `total_price` по позициям — работа `OrderService` (шаг `09`),
> запись в БД — работа `OrderRepository` (шаг `07`). Модель остаётся «только данными».
