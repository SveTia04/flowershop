# Шаг 00. Обзор проекта: что и зачем мы строим

> **Цель шага:** понять, какую систему мы делаем, какие проблемы она решает и кто ею
> пользуется. Кода здесь нет — это «карта местности» перед дорогой. Не пропускайте: без
> понимания «зачем» весь дальнейший код будет набором непонятных функций.

---

## 1. Простыми словами

Есть цветочный магазин. Сейчас учёт ведётся «на коленке»: где-то в тетради, где-то в
голове продавца, где-то в чате. Из-за этого:

- никто точно не знает, сколько каких цветов осталось;
- два сотрудника могут случайно оформить **один и тот же заказ дважды**;
- цветы — товар скоропортящийся, и их часто **списывают (выбрасывают)**, потому что
  забыли про срок годности;
- владелец не может быстро узнать, **сколько заработал** и **что продаётся лучше всего**.

Мы строим программу, которая всё это берёт на себя: единая база, роли сотрудников,
автоматический учёт остатков и сроков годности, отчёты.

Это **ровно то, что написано в Техническом задании** (раздел 2 «Спецификация проблемы»).

## 3. Цели системы (из ТЗ, раздел 3.2)

- уменьшить затраты и увеличить прибыль магазина;
- оптимизировать учёт и реализацию цветов;
- **исключить «замораживание денег»** — не закупать лишнее, прогнозировать спрос;
- меньше списаний просрочки.

Для нас, разработчиков, это превращается в конкретные технические задачи: учёт остатков,
сроки годности, авто-списание, отчёты. Их мы и реализуем.

## 4. Кто пользователи системы (роли)

В ТЗ (раздел 3.3) описаны роли. Запомните их — вокруг них строится **вся** логика прав
доступа (шаг `11`).

| Роль | Кто это | Что главное делает |
|------|---------|--------------------|
| **Администратор** | технический управляющий | управляет ролями, правами, справочниками, смотрит логи, блокирует пользователей |
| **Флорист** | собирает букеты | видит заказы и состав букета, отмечает этапы, списывает материалы |
| **Продавец** | работает с клиентами | оформляет продажи и возвраты, ведёт клиентов, следит за статусом заказа |
| **Курьер** | доставляет | видит список доставок и адреса, меняет статус доставки |
| **Закупщик / склад** | принимает товар | регистрирует поступления, оценивает качество, ставит срок годности |
| **Клиент** | покупатель | смотрит каталог, оформляет заказ, платит, отслеживает статус, оценивает |
| **Владелец** | собственник | смотрит отчёты по продажам, прибыли, популярным товарам |

> **Важная мысль для новичка:** одна и та же система показывает **разным людям разные
> возможности**. Это не семь программ — это одна программа с проверкой «а можно ли тебе
> это делать». Такой подход называется **RBAC** (управление доступом на основе ролей),
> подробно в шаге `11`.

## 5. Что именно умеет система (функции верхнего уровня)

Сгруппируем функции из ТЗ в «модули». Каждый модуль — это будущая часть кода.

- **Пользователи и доступ:** регистрация, вход, роли, права, блокировка, логи действий.
- **Каталог и товары:** цветы, букеты, состав букета, категории, цены.
- **Склад:** поступления, остатки, качество, сроки годности, авто-списание, уведомления
  о нехватке.
- **Заказы:** оформление (с клиентом или без), позиции заказа, расчёт цены, статусы,
  возврат/отмена, защита от дублей.
- **Доставка:** список доставок, адреса, статусы доставки.
- **Оплата:** способы оплаты, онлайн-оплата, подтверждение.
- **Уведомления:** о новом заказе, о дубле, об оплате, о доставке, о нехватке товара.
- **Отчёты:** продажи, прибыль, популярные товары.

Каждый модуль получит свои таблицы в БД (шаг `04`), свои модели (шаг `06`), репозитории
(шаг `07`) и сервисы (шаг `09`).

## 6. Границы проекта (что мы НЕ делаем)

Чтобы не утонуть, честно очертим границы. Это нормальная инженерная практика.

- **Графический интерфейс (окна/кнопки)** мы **не** делаем на C++. Наш C++ — это
  **сервер с REST API**. Интерфейс (сайт/приложение) — отдельная история; мы проверяем
  API через Postman или curl. Почему так — объяснено в шаге `08`.
- **Реальная онлайн-оплата** (банк, эквайринг) — заменяем «заглушкой» (имитацией успешной
  оплаты). Подключение настоящего платёжного шлюза выходит за рамки учебного проекта,
  но мы оставляем для него «место» в коде.
- **Прогнозирование спроса** (цель из ТЗ) реализуем в простом виде: отчёт о популярности
  + предупреждение о нехватке. Полноценное ML-прогнозирование — не для первого проекта.

> Указывать границы — признак зрелого инженера. Заказчику честнее сказать «это будет
> заглушка», чем обещать невозможное.

### 6.1. Куда расти дальше («будущее» проекта)

Этот проект сделан **практичным, а не идеальным** — его цель показать принципы C++ и
инженерии на живом примере, а не выпустить промышленный продукт. Это нормально: умение
сказать «вот здесь упрощённо, и вот почему» — само по себе важный инженерный навык.

Когда основа уложится в голове, проект можно развивать. Это **задел на будущее, делать
прямо сейчас не нужно** — но полезно видеть направление:

- **Безопасность паролей.** Заменить учебный хеш SHA-256 на «медленный» алгоритм
  **bcrypt** или **argon2** (см. шаг `10`, там это уже честно оговорено).
- **Промышленная БД.** Перейти с SQLite на **PostgreSQL**. Благодаря слоистой архитектуре
  (шаг `03`) меняется **только нижний слой** репозиториев — сервисы и API трогать не нужно.
  Это и есть награда за разделение на слои.
- **Деньги в копейках.** Сейчас цены хранятся в `double` (для простоты). В реальных
  финансовых системах деньги хранят в **целых копейках** (`int64_t`), чтобы не было ошибок
  округления.
- **Настоящая онлайн-оплата.** Подключить реальный платёжный шлюз вместо заглушки —
  «место» для него в коде уже оставлено.
- **Фронтенд.** Добавить простой **сайт или приложение** поверх готового REST API. API уже
  есть — осталась «витрина».
- **Прогнозирование спроса.** Развить простое предупреждение о нехватке до настоящего
  прогноза (статистика по продажам, сезонность).

> Главная мысль: хорошая архитектура **не закрывает дорогу будущему**, а оставляет её
> открытой. Каждый пункт выше можно добавить, не переписывая всю систему заново.

## 7. Как выглядит путь одного заказа (чтобы почувствовать систему)

Это «сквозной сценарий». Мы вернёмся к нему много раз.

```
1. Клиент открывает каталог букетов            → модуль Каталог
2. Клиент оформляет заказ, выбирает доставку    → модуль Заказы
3. Система проверяет: нет ли дубля? хватает ли
   цветов на складе? считает цену               → Сервис заказов + Склад
4. Клиент оплачивает (заглушка)                 → модуль Оплата
5. Продавцу и флористу приходит уведомление      → модуль Уведомления
6. Флорист собирает букет, списывает цветы       → модуль Склад
7. Курьер забирает заказ, везёт, меняет статус   → модуль Доставка
8. Клиент получает букет и ставит оценку         → модуль Заказы (рейтинг)
9. Владелец видит этот заказ в отчёте о продажах → модуль Отчёты
```

Если вы понимаете этот путь — вы понимаете проект. Весь код существует, чтобы эти 9
шагов происходили надёжно и без ошибок.

## 8. Нефункциональные требования (из ТЗ, разделы 3.4–3.6)

Это требования не к функциям, а к «качествам» системы:

- **Безопасность (ГОСТ 17799):** вход по паролю, контроль доступа, шифрование данных,
  защита передачи. → шаг `10`.
- **Надёжность:** система должна переживать сбои и восстанавливаться. Мы используем
  транзакции БД и аккуратную обработку ошибок. → шаги `07`, `15`.
- **Язык интерфейса — только русский.** → все сообщения и поля делаем на русском.
- **Хранение данных — реляционная СУБД.** → SQLite (учебно), путь к PostgreSQL — шаг `15`.
