# Приложение. Карта требований ТЗ → проект (трассировка)

> **Цель этого документа:** показать заказчику и преподавателю, что **каждый
> пункт Технического задания** учтён в проекте, и **где именно** он реализован — в каком слое,
> классе, таблице, и в каком шаге документации это описано. Это «карта трассировки требований»:
> по ней проверяют, что ничего не забыто, и видят, что **сознательно упрощено**.

> Колонка **Статус**: «реализовано» — сделано по существу; «упрощено» — суть выполнена, но
> проще промышленного варианта (честно отмечено); «заглушка» — обозначено, но не доведено
> (намеренно вынесено за рамки учебного проекта). Имена таблиц/полей/классов — строго из
> шагов `04`, `06`, `07`, `09`, `11`, `13`.

## 4. Раздел 3.4 ТЗ — Безопасность (ГОСТ/ISO 17799)

| Требование ТЗ | Где (слой/класс/таблица) | Шаг | Статус |
|----------------|---------------------------|-----|--------|
| Авторизация и контроль доступа | RBAC: `AuthorizationService::hasPermission`, `requirePermission` → `403`; `roles`, `permissions`, `role_permissions` | `11` | реализовано |
| Аутентификация (вход) | `AuthService` + токен `Authorization: Bearer` | `10` | реализовано |
| Хранение паролей защищённо | Хеш SHA-256 + соль; `users.password_hash`, `password_salt` | `10` | упрощено |
| Шифрование чувствительных данных в БД | Хеш паролей; (полное шифрование столбцов — расширение) | `10` | упрощено |
| Защита передачи данных | TLS через обратный прокси nginx (HTTPS) | `15` | реализовано |
| Защита от SQL-инъекций | Prepared statements (`?` + `bind`), без склейки строк | `07`, `13` | реализовано |
| Журнал действий (аудит) | `audit_log` | `12` | реализовано |

> **Честная оговорка про пароли.** В проде пароли хешируют **bcrypt/argon2** (медленные,
> устойчивые к перебору), а не SHA-256. Учебно мы используем SHA-256 + соль и **прямо об этом
> предупреждаем** (шаг `10`, спецификация). Статус — «упрощено».

## 9. Честный список упрощений и заглушек

Чтобы перед заказчиком/преподавателем не было сюрпризов — прямо перечисляем, что сделано
**проще** промышленного варианта или **обозначено, но не доведено**, и **почему**.

| Что | Как у нас | Как «по-взрослому» | Почему упростили |
|-----|-----------|---------------------|-------------------|
| Хеш паролей | SHA-256 + соль | bcrypt/argon2 | Учебно; принцип «хеш+соль» тот же, алгоритм проще объяснить |
| Деньги | `double` | целые копейки (`int64_t`) | Меньше отвлекающего кода; про риск округления честно сказано (шаги `09`, `13`) |
| Прогнозирование спроса | Сигнал «популярно + мало на складе» | Статистика/ML, тренды, сезонность | ML вне рамок учебного C++-проекта; наш сигнал реально полезен закупщику |
| Себестоимость для прибыли | Средняя закупочная цена `AVG(purchase_price)` | Точный учёт партии (FIFO/FEFO) | Точная себестоимость — большая отдельная тема |
| Онлайн-оплата | Таблица `payments` с методом `онлайн`, без интеграции с платёжным шлюзом | Интеграция с банком/эквайрингом | Внешние платёжные системы вне рамок проекта |
| Доставка (назначение курьера) | Таблица `deliveries`, статусы; `DeliveryService::assign` — как расширение | Полный модуль логистики | Показан образец, доведение оставлено как упражнение |
| Графический интерфейс (GUI) | Нет — только REST API | Веб-/десктоп-/мобильный клиент | По решению проекта делаем **только сервер** (шаг `08`); UI — отдельная задача |
| Шифрование данных в БД | Только пароли (хеш) | Шифрование чувствительных столбцов/файла БД | Достаточно для учебной цели; отмечено как расширение |
| Авто-списание по расписанию | Метод `writeOffExpired` есть; запуск по расписанию — внешним планировщиком | Встроенный планировщик/демон | Проще опереться на cron/Task Scheduler (шаг `15`) |
| Доступ к отчётам | право `report.view` (owner/admin) | то же | реализовано без упрощений |

> **Главный вывод для приёмки.** Все **функциональные** требования ТЗ по ролям (раздел 3.3)
> и все **проблемы** раздела 2 — закрыты по существу. Упрощения касаются «промышленной
> обвязки» (алгоритм хеша, тип денег, ML-прогноз, платёжный шлюз, GUI) и **явно
> задокументированы** — это нормально и ожидаемо для учебного проекта по ГОСТ 34.602-89.
