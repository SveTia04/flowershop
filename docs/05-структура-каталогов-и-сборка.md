# Шаг 05. Структура каталогов и сборка проекта

> **Цель шага:** создать **скелет** проекта — все папки в правильных местах — и написать
> один полноценный `CMakeLists.txt`, который собирает SQLite, подключает библиотеки из
> `third_party/` и строит исполняемый файл сервера и отдельную цель для тестов. В конце вы
> соберёте и запустите программу, которая печатает «Сервер цветочного магазина». Логики
> внутри пока нет — это **пустой, но честно собирающийся каркас**, на который мы дальше
> будем наращивать код.

> Перед этим шагом у вас уже должны быть установлены компилятор и CMake (шаг `01`) и лежать
> четыре библиотеки в `third_party/` (тоже шаг `01`). Если нет — вернитесь и доделайте.

## 1. Зачем вообще «структура каталогов»

Можно свалить все файлы в одну папку. Но уже на 20 файлах вы перестанете что-либо находить,
а компилятор начнёт путаться. Правильная структура — это **физическое воплощение
архитектуры** из шага `03`: у каждого **слоя** своя папка. Тогда вопрос «где лежит SQL по
заказам?» имеет мгновенный ответ: «в `repositories/order_repository.cpp`».

> Аналогия. Это как раскладка инструментов в мастерской: отвёртки — в одном ящике, ключи —
> в другом. Не потому что «так красиво», а потому что нужный инструмент находится за
> секунду.

## 2. Полная структура каталогов проекта

Создаём вот **ровно такую** структуру (она зафиксирована в общей спецификации проекта):

```
flowershop/
  CMakeLists.txt          ← описание сборки (главный файл этого шага)
  third_party/            ← чужой код: httplib.h json.hpp doctest.h sqlite3.c sqlite3.h
  db/                     ← SQL-файлы базы данных
    schema.sql            ←   создание таблиц (шаг 04)
    seed.sql              ←   тестовые данные (шаг 04)
  src/                    ← ВЕСЬ наш код на C++
    main.cpp              ←   точка входа: запуск сервера
    domain/               ← СЛОЙ МОДЕЛЕЙ: структуры-данные (шаг 06)
      role.h user.h product.h order.h stock.h delivery.h payment.h
      notification.h review.h supplier.h supply.h category.h
      flower_type.h audit_log.h order_status.h
    infra/                ← ИНФРАСТРУКТУРА: БД, конфиг, логи, безопасность
      database.h database.cpp
      config.h   config.cpp
      logger.h   logger.cpp
      security.h security.cpp
    repositories/         ← СЛОЙ ДОСТУПА К ДАННЫМ: SQL (шаг 07)
      user_repository.h user_repository.cpp
      order_repository.h order_repository.cpp
      product_repository.h product_repository.cpp
      stock_repository.h stock_repository.cpp
      ... (по одному на модуль)
    services/             ← СЛОЙ БИЗНЕС-ЛОГИКИ (шаг 09)
      auth_service.h auth_service.cpp
      order_service.h order_service.cpp
      stock_service.h stock_service.cpp
      report_service.h report_service.cpp
      notification_service.h notification_service.cpp
      pricing_service.h pricing_service.cpp
    api/                  ← СЛОЙ API: HTTP-маршруты, JSON (шаг 08)
      server.h server.cpp
      user_controller.h user_controller.cpp
      order_controller.h order_controller.cpp
      product_controller.h product_controller.cpp
      report_controller.h report_controller.cpp
      ...
  tests/                  ← ТЕСТЫ (шаг 14)
    test_main.cpp
    test_order_service.cpp
    ...
```

### 2.1. Назначение каждой папки

| Папка | Что хранит | Какой слой архитектуры (шаг 03) |
|-------|-----------|----------------------------------|
| `third_party/` | **чужой** код (библиотеки) | — (не наш код, отдельно) |
| `db/` | SQL-скрипты: создание таблиц и тестовые данные | сама БД |
| `src/` | **весь наш** C++-код | все слои |
| `src/domain/` | структуры-данные: `User`, `Order`, `Product`... | **Модели** |
| `src/infra/` | подключение к БД, конфиг, логи, хеш паролей | **Инфраструктура** |
| `src/repositories/` | SQL-запросы: достать/сохранить строки | **Репозитории** |
| `src/services/` | бизнес-правила магазина | **Сервисы** |
| `src/api/` | приём HTTP-запросов, разбор JSON, маршруты | **API** |
| `tests/` | автоматические проверки кода | — (проверяет все слои) |

> **Связь со слоями из шага 03.** Папки `api → services → repositories → domain/infra`
> повторяют «этажи» архитектуры. Правило «слой обращается только к нижнему слою» теперь
> ещё и физическое: файл из `api/` подключает заголовки из `services/`, файл из `services/`
> — из `repositories/`, и так вниз. Если вы вдруг увидите `#include` из `api/` внутри файла
> `repositories/` — это сигнал, что архитектуру нарушили.

### 2.2. Команды для создания структуры

**Linux / macOS** (в терминале bash/zsh):

```bash
# Идём в корень проекта (создан в шаге 01)
cd flowershop

# Создаём все папки сразу. -p создаёт вложенные и не ругается на существующие.
mkdir -p db src/domain src/infra src/repositories src/services src/api tests

# Создаём пустые файлы-заготовки, которые нам понадобятся прямо сейчас.
# (остальные файлы из структуры появятся на своих шагах)
touch CMakeLists.txt src/main.cpp tests/test_main.cpp db/schema.sql db/seed.sql
```

**Windows** (в PowerShell):

```powershell
cd flowershop

# Создаём папки (PowerShell делает вложенные сам, -Force не ругается на существующие)
mkdir db, src\domain, src\infra, src\repositories, src\services, src\api, tests -Force

# Создаём пустые файлы
New-Item CMakeLists.txt, src\main.cpp, tests\test_main.cpp, db\schema.sql, db\seed.sql -ItemType File -Force
```

## 3. Что такое CMake, target и «цель сборки»

Прежде чем писать `CMakeLists.txt`, разберём три слова.

- **CMake** — «генератор сборки». Он сам по себе **не компилирует**. Он читает ваш
  `CMakeLists.txt` и **готовит инструкции** для настоящего инструмента сборки вашей системы
  (Makefiles на Linux, проект Visual Studio на Windows и т.п.). Потом этот инструмент уже
  вызывает компилятор.
- **Target (цель)** — это **то, что нужно собрать**: либо **исполняемый файл** (программа),
  либо **библиотека**. В нашем проекте будет несколько целей: программа-сервер
  `flowershop`, цель тестов `flowershop_tests`, и (внутренне) библиотека из кода SQLite.
- **«Цель сборки»** в разговоре — это и есть target: «собери цель `flowershop`».

> Аналогия. `CMakeLists.txt` — это **проектная документация дома** (что и из чего строить).
> CMake — **прораб**, который по документации составляет наряд для бригады. Бригада
> (компилятор) уже кладёт кирпичи. Вы пишете документацию один раз — а строиться дом будет
> хоть в России, хоть в Канаде (на любой ОС).

## 4. Пишем полный `CMakeLists.txt`

Откройте файл `flowershop/CMakeLists.txt` и впишите следующее. Здесь каждая строка
прокомментирована — читайте комментарии, это и есть объяснение.

```cmake
# ── Базовая настройка проекта ──────────────────────────────────────────────

# Минимальная версия CMake. 3.16 есть везде и умеет всё, что нам нужно.
cmake_minimum_required(VERSION 3.16)

# Имя проекта и язык. CXX = C++, C нужен потому что sqlite3 написан на чистом C.
project(flowershop LANGUAGES CXX C)

# Требуем стандарт C++17 для всего проекта и запрещаем "тихо откатиться" на старый.
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)   # без нестандартных расширений компилятора — чистый C++17

# Куда складывать готовые программы — в папку build/bin (чтобы не искать по подпапкам).
set(CMAKE_RUNTIME_OUTPUT_DIRECTORY ${CMAKE_BINARY_DIR}/bin)


# ── Зависимость 1: SQLite (компилируем из исходника на C) ───────────────────

# sqlite3 поставляется как ОДИН .c-файл. Делаем из него отдельную статическую
# библиотеку-цель "sqlite3". Так её код соберётся один раз и подключится куда нужно.
add_library(sqlite3 STATIC third_party/sqlite3.c)

# Кто подключит цель sqlite3 — автоматически увидит заголовок sqlite3.h из third_party.
target_include_directories(sqlite3 PUBLIC ${CMAKE_SOURCE_DIR}/third_party)


# ── Header-only зависимости (httplib.h, json.hpp, doctest.h) ────────────────

# Эти библиотеки — просто заголовки, собирать нечего. Достаточно, чтобы компилятор
# знал, ГДЕ их искать. Заведём "интерфейсную" цель, несущую только путь к заголовкам.
add_library(third_party_headers INTERFACE)
target_include_directories(third_party_headers INTERFACE ${CMAKE_SOURCE_DIR}/third_party)


# ── Наш код: соберём его в библиотеку, чтобы переиспользовать в сервере и тестах ──

# Собираем список ВСЕХ наших .cpp из src/, КРОМЕ main.cpp (его подключим только к серверу).
# GLOB_RECURSE рекурсивно ищет файлы. CONFIGURE_DEPENDS — пере-сканировать при изменениях.
file(GLOB_RECURSE FS_SOURCES CONFIGURE_DEPENDS ${CMAKE_SOURCE_DIR}/src/*.cpp)
list(REMOVE_ITEM FS_SOURCES ${CMAKE_SOURCE_DIR}/src/main.cpp)

# Если кода ещё нет (а на этом шаге его почти нет) — создаём библиотеку с заглушкой,
# чтобы CMake не ругался на "библиотеку без файлов".
if(FS_SOURCES STREQUAL "")
    # ${CMAKE_BINARY_DIR} — папка сборки; пишем туда крошечный пустой .cpp.
    file(WRITE ${CMAKE_BINARY_DIR}/_empty.cpp "// заглушка: код появится на следующих шагах\n")
    set(FS_SOURCES ${CMAKE_BINARY_DIR}/_empty.cpp)
endif()

# Библиотека flowershop_lib — это весь наш код проекта (без main).
add_library(flowershop_lib STATIC ${FS_SOURCES})

# Эта библиотека видит свои же заголовки (папки src/*) и зависит от sqlite3 и заголовков.
target_include_directories(flowershop_lib PUBLIC ${CMAKE_SOURCE_DIR}/src)
target_link_libraries(flowershop_lib PUBLIC sqlite3 third_party_headers)


# ── Цель 1: исполняемый файл сервера ────────────────────────────────────────

# Программа flowershop собирается из main.cpp и линкуется с нашей библиотекой кода.
add_executable(flowershop src/main.cpp)
target_link_libraries(flowershop PRIVATE flowershop_lib)


# ── Цель 2: тесты (отдельная программа) ─────────────────────────────────────

# Собираем все .cpp из tests/. Тесты линкуются с тем же кодом проекта (flowershop_lib),
# поэтому проверяют РЕАЛЬНЫЙ код, а не его копию.
file(GLOB_RECURSE FS_TEST_SOURCES CONFIGURE_DEPENDS ${CMAKE_SOURCE_DIR}/tests/*.cpp)

if(NOT FS_TEST_SOURCES STREQUAL "")
    add_executable(flowershop_tests ${FS_TEST_SOURCES})
    # doctest лежит в third_party — даём цели тестов путь к заголовкам.
    target_link_libraries(flowershop_tests PRIVATE flowershop_lib third_party_headers)

    # Подключаем встроенный в CMake механизм тестов, чтобы можно было запускать "ctest".
    enable_testing()
    add_test(NAME all_tests COMMAND flowershop_tests)
endif()
```

> **Почему наш код вынесен в библиотеку `flowershop_lib`, а не собран прямо в `flowershop`?**
> Потому что тесты (`flowershop_tests`) должны проверять **тот же самый** код. Если бы код
> жил только внутри программы-сервера, тестам пришлось бы его дублировать. Общая библиотека
> решает это: и сервер, и тесты линкуются с ней. Это прямое следствие принципа
> «тестируемости» из шага `03`.

> **Почему `main.cpp` исключён из библиотеки?** В программе может быть только одна функция
> `main`. У сервера свой `main` (`src/main.cpp`), у тестов — свой (внутри `tests/`,
> сгенерированный doctest). Если бы оба `main` попали в одну сборку — линковщик выдал бы
> «multiple definition of main».

## 5. Минимальный `main.cpp`

Теперь — заглушка сервера. Откройте `src/main.cpp`:

```cpp
// src/main.cpp — точка входа сервера цветочного магазина.
// На этом шаге логики ещё нет: мы только проверяем, что весь каркас собирается
// и запускается. Настоящий запуск HTTP-сервера появится в шаге 08.

#include <iostream>   // для вывода в терминал

// main — точка входа программы (см. шаг 02). int = вернём код завершения.
int main() {
    // Печатаем приветствие. На русском — потому что язык интерфейса проекта русский
    // (нефункциональное требование ТЗ, см. шаг 00).
    std::cout << "Сервер цветочного магазина" << std::endl;

    // 0 = программа завершилась успешно.
    return 0;
}
```

> На этом шаге `main.cpp` ещё **не** подключает ни SQLite, ни httplib — нам важно сперва
> убедиться, что сама **машинерия сборки** работает. Библиотеки начнут использоваться с
> шага `07`.

## 6. Собираем и запускаем каркас

Команды CMake одинаковы на всех ОС (в этом его сила). Находясь в `flowershop/`:

```bash
# Шаг 1 — конфигурация: CMake читает CMakeLists.txt и готовит сборку в папке build.
cmake -B build

# Шаг 2 — сборка: компилируется sqlite3 и наш main.cpp, линкуется программа flowershop.
cmake --build build
```

Запуск получившейся программы:

```bash
# Linux / macOS:
./build/bin/flowershop

# Windows (PowerShell). Папка конфигурации может отличаться (Debug/Release):
# .\build\bin\flowershop.exe
# либо .\build\bin\Debug\flowershop.exe
```

Ожидаемый вывод:

```
Сервер цветочного магазина
```

Чтобы прогнать тесты (когда они появятся в шаге `14`), есть встроенная команда:

```bash
# Запускает все зарегистрированные тесты из папки build.
ctest --test-dir build
```

> Если что-то не собралось — читайте **первую** ошибку (см. раздел частых ошибок в шаге
> `01`). Самые вероятные на этом шаге: не на месте `third_party/sqlite3.c` (CMake не найдёт
> файл при `add_library(sqlite3 ...)`) или запуск `cmake` не из папки `flowershop/`.

## 7. Что мы получили и как это связано с дальнейшим

```
flowershop/
  CMakeLists.txt   ← собирает: sqlite3 + наш код (lib) → сервер + тесты
  build/           ← здесь живёт результат сборки (эту папку в git не кладут)
  src/main.cpp     ← печатает "Сервер цветочного магазина"
  src/domain/      ← (пусто) сюда придут модели в шаге 06
  src/repositories ← (пусто) сюда придёт SQL в шаге 07
  ...
```

Каркас **расширяемый**: благодаря `GLOB_RECURSE` любой новый `.cpp`, который вы положите в
`src/` на следующих шагах, **автоматически** попадёт в сборку — `CMakeLists.txt` менять не
придётся (нужно лишь иногда заново запустить `cmake -B build`, чтобы он пере-сканировал
файлы; `CONFIGURE_DEPENDS` старается делать это сам).

Дальнейшая дорога: в шаге `06` мы наполним `src/domain/` структурами-моделями (`User`,
`Order`, `Product`...), и они тут же станут частью этой сборки.

> **Совет про git.** Папку `build/` в репозиторий не кладут — это результат сборки, его
> всегда можно пересобрать. Создайте файл `.gitignore` в корне со строкой `build/`. Папки
> `src/`, `db/`, `third_party/` и `CMakeLists.txt` — кладём в git обязательно.
